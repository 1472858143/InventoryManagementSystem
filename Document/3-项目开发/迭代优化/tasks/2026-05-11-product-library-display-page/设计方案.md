# 设计方案

## 问题定义

当前商品库页面 (`frontend/src/views/product/ProductView.vue`) 以传统数据表格形式展示商品，存在以下问题：

- 表格信息密度高但视觉单调，缺乏现代化后台管理系统应有的质感和层级。
- 商品状态仅以 `el-tag` 表达，识别度弱，扫视时不易快速判断商品是否上架。
- 商品详情字段散落在表格列中，未来若新增折扣、促销、库存联动等扩展模块缺乏承载位。
- 没有搜索、筛选、排序能力，商品多时检索成本高。
- 新增商品表单要求手工填写「商品编号」，对实际超市运营无业务价值，是登记摩擦点。

本任务对商品库页面进行整体展示优化，将「展示形态、详情承载、检索能力、表单流程」四件事统一升级。

## 现状分析

- **当前前端行为**：`ProductView.vue` 单文件包含表格、新增弹窗、状态切换三段逻辑；列字段为商品编号、商品名称、分类、单位、进价、售价、状态、创建时间、操作。无搜索、筛选、排序。
- **当前后端行为**：
  - `GET /api/products` 返回 `ProductListItemResponse`（id, productCode, productName, categoryId, categoryName, unit, purchasePrice, salePrice, status, createTime），不含销量。
  - `POST /api/products` 接收 `ProductCreateRequest`，要求 productCode 必填且唯一。
  - `PUT /api/products/status` 切换 0/1 状态。
- **当前数据库结构**：`product` 表 `product_code VARCHAR(50) NOT NULL` + `UNIQUE KEY uk_product_code`；`stock_log` 表 `change_type` 含 INBOUND/OUTBOUND/CHECK/SHELF_STATUS（IT-002 后）。无独立销售明细表，销量需要从 `stock_log.change_type = 'OUTBOUND'` 派生。
- **当前文档描述**：`Document/开发记录/frontend/前端开发规划.md` 要求 views 与后端模块对应；商品页面只承担基础数据展示与接口调用，不承担库存计算职责。

## 方案选择

### 方案 A：原地升级 ProductView.vue + 子组件拆分（推荐）

- **内容**：
  - `ProductView.vue` 保留为页面容器，承担数据加载、状态、子组件协调；表格全部移除。
  - 新增 `frontend/src/views/product/components/ProductCard.vue`：单个商品卡片。
  - 新增 `frontend/src/views/product/components/ProductDetailDrawer.vue`：商品详情抽屉。
  - 新增 `frontend/src/views/product/components/ProductFilterBar.vue`：搜索 + 筛选 + 排序的横向工具栏。
  - 后端 `ProductListItemResponse` 增加 `salesCount` 字段；`ProductMapper.xml` 在 findAllWithCategory 上 LEFT JOIN 一段聚合子查询。
  - 后端 `ProductCreateRequest.productCode` 改为可选；Service 层在缺省时自动生成（如 `P` + `yyyyMMddHHmmss` + 4 位随机数字），保留唯一索引。
- **优点**：
  - 与 IT-002 已建立的设计 token、Lucide 图标体系无缝衔接。
  - 拆分清晰：主页面只管数据流；子组件只管展示与交互；详情抽屉独立扩展。
  - 后端最小改动：只动 1 个 SQL + 1 个 VO + 1 个 Service 方法，不新增接口。
  - 前端检索能力为纯前端实现，毕设演示数据量足够支撑，避免后端引入分页/查询参数的复杂度。
- **风险**：
  - 商品数据量过大时（>500 条）前端排序/筛选开始有感延迟。当前毕设场景可控。
  - 中文「首字母排序」需引入 `pinyin-pro` 依赖（约 30KB gzipped），或自实现简易首字母映射。

### 方案 B：保留表格 + 卡片视图切换

- **内容**：在表格上方加一个 `卡片视图 / 列表视图` 切换器，二者共存。
- **优点**：保留旧用户习惯。
- **风险**：与需求 1「卡片形式替代传统表格形式」矛盾；维护两套视图增加开发与回归成本；用户决策疲劳。

### 方案 C：商品详情独立路由页面

- **内容**：详情走 `/product/:id` 独立路由。
- **优点**：详情页空间不受限。
- **风险**：与需求 10「悬浮层形式」、11「不使用新页面跳转」直接冲突。

## 推荐方案

采用 **方案 A**。

### 卡片设计（需求 1-5、13）

- **网格布局**：CSS Grid `repeat(auto-fill, minmax(240px, 1fr))`，间距 16px。
- **卡片尺寸**：宽度自适应、最小 240px；高度由内容决定，约 160px。
- **卡片结构**（自上而下）：
  1. 卡片顶部：商品名称（16px / semi-bold）+ 状态文字徽章（右上角）。
  2. 中部：分类标签（小 chip）+ 单位。
  3. 底部：售价（强调）+ 进价（次要灰色）。
- **状态视觉**：
  - 上架：左侧 4px 实心边条 `--color-success` (#10B981) + 状态文字「上架」绿色。
  - 下架：左侧 4px 实心边条 `--color-text-secondary` (#94A3B8) + 状态文字「下架」灰色 + 卡片整体 opacity 0.7。
- **交互**：整张卡片为可点击区域（hover 时阴影加深、轻微上浮 2px），点击触发详情抽屉。

### 详情抽屉设计（需求 6-12）

- **容器**：`el-drawer`，方向 `rtl`，宽度 520px。
- **内部分区结构**（由上到下，每个分区独立 card）：
  1. **商品信息分区**：商品名称（标题级）、商品编号（只读小字，让用户知道系统已自动生成）、分类、单位、进价、售价、创建时间。
  2. **状态管理分区**：当前状态徽章 + 「上架 / 下架」切换按钮 + 操作反馈区。
  3. **扩展占位分区**（预留，本期不实现，但写注释说明）：折扣管理、促销活动、关联库存、操作历史。占位采用「即将上线」灰色卡片或直接以注释形式预留挂载点。
- **拆分约束**（需求 12）：详情抽屉以独立组件 `ProductDetailDrawer.vue` 实现，`ProductView.vue` 仅通过 v-model:visible + props:product 控制；详情内的状态变更通过 emit 通知主页面刷新。

### 工具栏设计（需求 16-23）

工具栏组件 `ProductFilterBar.vue` 横向布局，自上而下：

- **第一行**：搜索输入框（宽 320px，左侧搜索图标，placeholder「搜索商品名称或分类」，300ms 防抖） + 排序下拉（宽 160px） + 「筛选」按钮（点击展开下方折叠区） + 「新增商品」按钮（右对齐）。
- **第二行（默认折叠，点开展示）**：售价区间（min - max）+ 进价区间（min - max）+ 分类多选下拉 + 状态单选（全部 / 上架 / 下架）+「重置」按钮。

#### 搜索逻辑（需求 16-17）

```
filtered = list.filter(p =>
  !q || p.productName.includes(q) || (p.categoryName ?? '').includes(q)
)
```

防抖 300ms，避免频繁重算。

#### 筛选逻辑（需求 18-20）

各条件之间为 AND 关系；空条件视为不约束：

- 售价范围：`salePrice >= minSalePrice && salePrice <= maxSalePrice`
- 进价范围：同上
- 分类：`selectedCategoryIds.length === 0 || selectedCategoryIds.includes(p.categoryId)`
- 状态：`statusFilter === 'all' || p.status === statusFilter`

#### 排序逻辑（需求 21-23）

排序键值 `sortKey`：

| sortKey | 含义 | 实现 |
| --- | --- | --- |
| `salesDesc` | 销量降序（默认） | `b.salesCount - a.salesCount` |
| `nameAsc` | 商品名首字母 A→Z | 取拼音首字母 localeCompare |
| `categoryAsc` | 分类名首字母 A→Z | 同上 |
| `priceAsc` | 售价低→高 | `a.salePrice - b.salePrice` |
| `priceDesc` | 售价高→低 | `b.salePrice - a.salePrice` |

中文首字母方案：引入 `pinyin-pro` 轻量依赖，调用 `pinyin(name, { pattern: 'first', toneType: 'none' })` 取首字母。

### 新增表单调整（需求 14-15）

- 移除「商品编号」表单项与对应 rules。
- 提交时 `createProduct` 不再传 `productCode`。
- 后端 `ProductServiceImpl.createProduct` 在 `request.productCode()` 为空时自动生成：
  - 格式：`P` + `yyyyMMddHHmmss` + 4 位随机数字（共 19 字符）。
  - 唯一性碰撞概率极低；如插入失败因唯一索引冲突，重试一次。
- `ProductCreateRequest.productCode` 字段改为可选；如果用户后续仍想传入则尊重传入值，不影响向下兼容。

### 销量数据来源

后端 `ProductMapper.xml` `findAllWithCategory` 加 LEFT JOIN：

```sql
LEFT JOIN (
  SELECT product_id, COALESCE(SUM(change_quantity), 0) AS sales_count
  FROM stock_log
  WHERE change_type = 'OUTBOUND'
  GROUP BY product_id
) s ON s.product_id = p.id
```

`ProductView` model 新增 `Integer salesCount`，`ProductListItemResponse` 新增字段，前端展示在卡片底部小字（可选，本期不强制）。

## 实施分阶段

设计落地后建议拆分 4 个阶段（具体由后续 plan 任务展开，本文档不输出代码）：

1. **P1 后端微调**：列表 SQL 加销量；createProduct 编号自动生成；DTO/VO 字段调整；后端单测覆盖。
2. **P2 前端基础组件**：ProductCard、ProductFilterBar、ProductDetailDrawer 三个独立组件先按 props/emit 契约搭好骨架。
3. **P3 ProductView 重构**：替换表格为卡片网格；接入搜索/筛选/排序计算属性；接入详情抽屉；新增弹窗去掉商品编号字段。
4. **P4 验证**：手动走查 22 项验收点；视觉与 IT-002 token 体系一致性检查；冒烟测试新增/上下架/详情切换流程。

## 影响边界

- **会修改**：
  - 前端 product 模块全部文件（ProductView 重写 + 3 个新增子组件）。
  - 前端 `package.json`（新增 pinyin-pro 依赖）。
  - 后端 product 模块的 ProductCreateRequest、ProductServiceImpl、ProductListItemResponse、ProductView model、ProductMapper.xml。
  - 文档：本任务目录全部文件、`00-问题清单.md` 中 IT-003 状态与备注。
- **不会修改**：
  - 数据库表结构（不改 product/stock_log/category 任何列）。
  - 正式 SQL `sql/market.sql` 与 `Document/3-项目开发/market.sql`。
  - 库存模块、报表模块、首页驾驶舱、登录认证模块。
  - 路由结构（详情走抽屉，不新增路由）。
- **已定细节**（2026-05-12 用户确认）：
  - 商品编号自动生成格式：`P` + `yyyyMMddHHmmss` + 4 位随机数字（共 19 字符）。
  - 卡片底部展示销量数字（小字，作为决策辅助信息）。
  - 详情抽屉扩展槽位：以注释预留挂载点，不渲染「即将上线」占位卡片。

## 回滚策略

- **前端回滚**：保留 git 历史，回退到本次重构前的 commit；卸载 pinyin-pro 依赖。
- **后端回滚**：
  - createProduct 校验恢复「商品编码不能为空」即可；DB 已生成的自动编号数据不影响业务（仍然是合法 productCode）。
  - ProductListItemResponse 移除 salesCount 字段；前端 v2 未上线时无消费方。
  - ProductMapper.xml 删除 LEFT JOIN 子查询。
- **数据回滚**：本任务不变更表结构，无 DB 回滚动作。
- **任务文档回滚**：删除本任务目录设计内容回到「现状分析」状态，或将整目录移入 `archive/`。
