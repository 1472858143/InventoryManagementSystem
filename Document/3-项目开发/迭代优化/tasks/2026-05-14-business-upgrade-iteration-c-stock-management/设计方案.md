# 设计方案

## 问题定义

IT-006 已将采购、销售和退货纳入统一库存变更入口，但库存管理仍缺少报损、报溢和库存报警能力。当前业务无法记录商品损耗、盘盈入库等库存调整原因，也无法在独立页面识别低于下限或高于上限的库存。

## 现状分析

- 当前前端行为：已有库存总览、库存工作台和上架补货等页面；尚无报损、报溢和报损报溢查询页面。
- 当前后端行为：已有 `StockChangeService`，支持 `STOCK_LOSS`、`STOCK_OVERFLOW` 枚举值，但尚无库存调整单据模块。
- 当前数据库结构：运行库已有 `stock`、`stock_log` 和 IT-006 的业务单据表；尚无 `stock_adjustment_order` 和 `stock_adjustment_order_item`。
- 当前文档描述：IT-004 设计稿将报损报溢和库存报警划入迭代 C。

## 方案选择

### 方案 A：用单张库存调整表记录商品行

- 内容：每条报损 / 报溢明细直接作为一条调整记录，不建主表。
- 优点：实现简单，少一张表。
- 风险：难以支持多商品单据、统一作废、单据号、总数量和查询详情；与采购 / 销售单据模型不一致。

### 方案 B：主表 + 明细表，复用统一库存变更入口

- 内容：新增 `stock_adjustment_order` 主表和 `stock_adjustment_order_item` 明细表；报损 / 报溢按一张单据多条明细保存。
- 优点：与 IT-006 单据模型一致；天然支持多商品、作废冲销、详情查询和审计追踪。
- 风险：实现量略大，需要前端复用明细编辑和详情查询能力。

## 推荐方案

采用方案 B。

原因：报损 / 报溢同样属于库存来源单据，应与采购、销售、退货采用一致的主表 + 明细表模型。库存变更继续通过 `StockChangeService`，以 `source_type=STOCK_LOSS / STOCK_OVERFLOW` 写入 `stock_log`，避免再次出现库存口径分散。

## 核心规则

- 报损：`type=LOSS`，每条明细调用 `StockChangeService`，`delta=-quantity`，`source_type=STOCK_LOSS`。
- 报溢：`type=OVERFLOW`，每条明细调用 `StockChangeService`，`delta=+quantity`，`source_type=STOCK_OVERFLOW`。
- 作废：`NORMAL -> CANCELLED`，逐条写入反向库存变更。
- 已作废单据不允许重复作废。
- 报损作废为库存增加；报溢作废为库存减少，若库存不足则阻止作废。
- 同一张库存调整单不允许重复选择商品。
- `unit_price`、`subtotal` 可为空；若前端提供单价，后端可计算小计。

## 查询与页面

- 报损表单页：`/stock/loss`
- 报溢表单页：`/stock/overflow`
- 报损报溢查询页：`/stock/adjustments`
- 库存报警页：`/stock/alarms`

查询支持：

- 类型：`LOSS` / `OVERFLOW`
- 单据号模糊查询
- 商品 ID
- 操作人 ID
- 创建日期区间
- 分页

库存报警：

- 低库存：`stock.quantity < stock.min_stock`
- 高库存：`stock.quantity > stock.max_stock`
- 不新增报警历史表，实时查询即可。

## 影响边界

- 会修改：库存相关后端模块、前端库存菜单和路由、任务 SQL 草案。
- 不会修改：正式 SQL、统计报表、权限系统、系统日志、旧入库 / 出库历史表。
- 需要用户确认：本任务验收后是否将 `stock_adjustment_order` 两张表同步到正式 SQL。

## 回滚策略

代码层面可移除新增库存调整模块、路由和菜单入口；数据库层面仅任务草案，不自动同步正式 SQL。若已在运行库执行 DDL，需按验收结果单独评估是否保留或清理测试表。
