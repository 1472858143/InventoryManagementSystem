# 前端影响

## 是否涉及前端

- 结论：涉及，但第一阶段只产出设计，不改前端代码。
- 影响目录：`frontend/src/layout/AdminLayout.vue`、`frontend/src/router/index.js`、views 下各业务页面目录、`frontend/src/api/` 下 API 封装。
- 影响组件：当前库存查询页可被多个入口复用，需在迭代 C 完成时抽成可复用查询组件 / 抽屉。
- 影响 API 封装：后续将新增 supplier / customer / purchase / purchase-return / sales / sales-return / stock-adjustment / system-log / role / permission 等 API 文件。

## 用户交互变化

升级后的菜单结构（详见 `设计方案.md` §1）：

```
首页
进货管理 / 销售管理 / 库存管理 / 统计报表 / 基础资料 / 系统管理
```

关键差异：

- 当前库存查询集中在"库存管理 → 当前库存"一处，不在进货 / 销售下重复挂载。其他业务页用抽屉组件复用查询能力。
- "报表分析"改造为"综合报表"并入"统计报表"，旧 ReportView 保留作为综合报表页。
- "商品分类管理"独立菜单下线，分类入口收纳进商品表单。
- 安全退出、密码修改不进菜单，保留在顶栏头像下拉。
- 菜单项绑定 permission_code，登录后由后端返回当前用户可见菜单与权限编码，前端动态渲染菜单 + 路由守卫。

## 数据契约变化

第一阶段不改请求参数和响应字段。后续按迭代落地：

- 迭代 A：`/api/suppliers`、`/api/customers`，商品 API 不变但分类下拉接口改为内嵌使用。
- 迭代 B：`/api/purchases`、`/api/purchase-returns`、`/api/sales`、`/api/sales-returns`，单据列表 + 单据明细两类返回结构。
- 迭代 C：`/api/stock-adjustments`、`/api/stock-alarms`（查询型）。
- 迭代 D：`/api/reports/purchase`、`/api/reports/sales`、`/api/reports/supplier`、`/api/reports/customer`。
- 迭代 E：登录响应扩展为 `{ user, menus[], permissionCodes[] }`；`/api/roles`、`/api/system-logs`、`/api/auth/password`。

## 兼容性风险

- 多个新菜单都包含库存查询的场景已通过"集中一处 + 抽屉复用"消解，需在迭代 C 落地复用组件，避免后续多入口各写一份。
- 菜单结构与原有截图、演示材料不一致，后续迭代结束时同步刷新演示截图与项目书材料。
- 菜单级权限上线后（迭代 E），路由守卫与 store 持久化的顺序要正确，否则会出现"刷新瞬间菜单空"的视觉问题。设计要点已写入迭代 E 风险登记。
- 旧"入库 / 出库"菜单下线后，相关旧路由要么 redirect 到新菜单首页，要么显式 404，不允许悬空。
