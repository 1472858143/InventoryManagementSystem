# 设计方案

## 问题定义

业务升级整体设计由 IT-004 完成。本任务承担迭代 A：建立基础资料模块（供应商、客户、商品菜单迁移）+ 为后续 system 类功能预建表 + 修复字段对账偏差。

## 现状分析

- 当前前端行为：商品管理与商品分类管理为独立菜单；无供应商 / 客户管理入口。
- 当前后端行为：product / category 模块齐备；无 supplier / customer 模块。
- 当前数据库结构：正式 SQL 已有 product / category / stock 等表，但缺 supplier / customer / system_log / permission / role_permission。同时存在字段偏差：`stock.shelf_status` 与 `stock_log.stock_type` 在代码已用但正式 SQL 缺失；`restock_order` Mapper 存在但表不存在。
- 当前文档描述：IT-004 设计稿已经定稿（commit `a372648`），本任务直接执行设计稿中"迭代 A · 基础资料"的范围。

## 方案选择

本任务**不重新决策**已在 IT-004 定稿的事项。所有数据模型、表字段、菜单结构、命名规范均沿用 IT-004 设计稿。

执行层面的小决策（如字段对账具体修复路径、SupplierService 与既有 ProductService 的依赖关系）在本会话 brainstorming 中再确定，并写入 `开发记录.md` 决策记录段。

### 沿用自 IT-004 的关键决策（不可改）

- supplier / customer 表结构（编码 + 名称 + 联系人 + 电话 + 地址 + 状态）。
- 商品分类菜单收纳到商品表单内部，不在基础资料独立挂菜单。
- system_log / permission / role_permission 仅建表，不接入业务逻辑。
- 不引入 StockChangeService。
- 不下线旧 inbound / outbound 菜单。

### 本任务内待确认的执行细节

- 字段对账具体修复路径（如 `restock_order` Mapper 删除还是建表）。
- supplier / customer 编码生成规则（手填 vs 自动生成）。
- supplier / customer 在被引用前不允许停用的校验是否本次落地（迭代 A 没有引用方，可以推迟到 B）。

## 推荐方案

执行 IT-004 设计稿"迭代 A"段。本任务的设计稿不再展开重复内容，仅指向 IT-004：

- 数据模型：见 IT-004 `设计方案.md` §2 表清单 + `数据库变更.sql` "迭代 A 落地"段。
- 菜单结构：见 IT-004 `设计方案.md` §1 基础资料组。
- 验收边界：见 IT-004 `迭代拆分.md` "迭代 A · 基础资料"段。

## 影响边界

- 会修改：本任务目录文件、`sql/market.sql`、`Document/3-项目开发/market.sql`、前端 `master/` 目录、后端 supplier / customer 模块。
- 不会修改：进货 / 销售 / 退货 / 报损报溢相关代码（不存在）、`StockChangeService`（不存在）、`stock_log` 业务字段（迭代 B）、正式项目书 / 需求分析 / 系统设计正文。
- 需要用户确认：字段对账具体修复策略、supplier / customer 编码生成规则。

## 回滚策略

- 数据库变更可回滚：DROP 5 张新表 + 恢复 `stock_log` 修改前字段定义。
- 前端可回滚：恢复 router 与 AdminLayout 的菜单结构；删除 master/ 下新增页面。
- 后端可回滚：删除 supplier / customer 模块；恢复字段对账涉及的 Mapper / Entity。
- 旧 product / category / stock 业务行为本任务不修改，无需回滚。
