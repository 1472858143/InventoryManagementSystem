# 后端影响

## 是否涉及后端

- 结论：涉及。
- 新增模块：`com.supermarket.inventory.supplier`、`com.supermarket.inventory.customer`。
- 影响 Controller：SupplierController（CRUD + 状态切换）、CustomerController（CRUD + 状态切换）。
- 影响 Service：SupplierService、CustomerService。
- 影响 Mapper：SupplierMapper、CustomerMapper。
- 影响实体或 DTO：Supplier / Customer Entity；SupplierRequest / SupplierVO / CustomerRequest / CustomerVO。
- 不引入 StockChangeService（留给迭代 B）。

## 业务逻辑变化

- supplier / customer 支持创建、修改、启停用（status 字段）。
- 编码（code）唯一约束在 Service 层校验冲突，提示友好错误。
- 启停用不级联检查引用（迭代 A 没有引用方）；引用校验留给迭代 B 落地。
- 字段对账修复后，既有 stock / stockLog / report 模块的查询逻辑应保持不变；若已有偏差掩盖的 bug 浮现，记录到 `开发记录.md` 问题段。

## 接口变化

新增接口：

- `GET /api/suppliers`（分页 / 关键字过滤）
- `POST /api/suppliers`
- `PUT /api/suppliers/{id}`
- `PATCH /api/suppliers/{id}/status`
- `GET /api/customers`、`POST /api/customers`、`PUT /api/customers/{id}`、`PATCH /api/customers/{id}/status`

不修改现有接口的契约。

## 数据一致性

- supplier / customer 编码唯一性靠数据库 UNIQUE KEY + Service 层校验双保险。
- 状态字段 status：1=启用 / 0=停用，软删除语义，不物理删除。
- system_log / permission / role_permission 在本任务**仅建表**，无任何写入或读取业务逻辑，避免迭代 A 越界。
- 字段对账：修复后须保证 stock / stockLog / report 既有接口的回归测试通过。

## 后端验证

- 单元测试：SupplierService、CustomerService 的编码唯一冲突、状态切换、分页查询用例。
- 集成测试：SupplierController、CustomerController 主路径。
- 手动接口验证：通过 Postman 或前端调用确认 CRUD 与启停用。
- 字段对账回归：跑既有 stock / report 测试套件，确认无退化。
