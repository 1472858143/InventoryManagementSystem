# 后端影响

## 是否涉及后端

- 结论：是，影响范围大，需新增和修改多个层级的代码
- 影响 Controller：StockController（新增补货接口）、CategoryController（新增，全量 CRUD）
- 影响 Service：StockService（入库/出库/补货逻辑拆分）、CategoryService（新增）
- 影响 Mapper：StockMapper（字段变更）、CategoryMapper（新增）、RestockOrderMapper（新增）
- 影响实体或 DTO：Stock、Product、新增 Category、RestockOrder

## 业务逻辑变化

### 入库（InboundService）
- **变更前**：`stock.quantity += inbound.quantity`
- **变更后**：`stock.warehouse_quantity += inbound.quantity`
- stock_log 写入时 `stock_type = 'WAREHOUSE'`

### 出库/购买（OutboundService）
- **变更前**：`stock.quantity -= outbound.quantity`；库存不足判断 `quantity < outbound.quantity`
- **变更后**：`stock.shelf_quantity -= outbound.quantity`；库存不足判断 `shelf_quantity < outbound.quantity`
- 若 `shelf_quantity` 不足需明确提示"上架库存不足，请先补货"
- stock_log 写入时 `stock_type = 'SHELF'`

### 补货（新增 RestockService）
- 原子事务：`warehouse_quantity -= N`，`shelf_quantity += N`
- 前置校验：`warehouse_quantity >= N`，否则抛出"仓库库存不足"异常
- 写入 `restock_order` 记录
- 写入两条 `stock_log`：RESTOCK/WAREHOUSE（减少）+ RESTOCK/SHELF（增加）

### 盘点（StockCheckService）
- `system_quantity = warehouse_quantity + shelf_quantity`（总量，已确认）
- `difference = actual_quantity - system_quantity`
- 盘点调整策略（已确认）：差值优先调整 `warehouse_quantity`
  - 正差值（实盘多于系统）：`warehouse_quantity += difference`
  - 负差值（实盘少于系统）：先从 `warehouse_quantity` 扣减，不足时再从 `shelf_quantity` 扣减
  - 最终保证 `warehouse_quantity >= 0` 且 `shelf_quantity >= 0`

### 分类（新增 CategoryService）
- 提供：新增分类、查询全部启用分类列表（供前端下拉）、禁用/启用分类
- 禁用分类时，若有商品正在使用该分类，需提示并拒绝，或允许禁用但已关联商品不受影响（推荐后者）

### 商品（ProductService）
- 新增商品：`category_id` 必填（从前端下拉选择），`unit` 必填有默认值
- 修改商品：同上
- 返回商品列表时，JOIN `category` 表返回 `category_name`，前端不再存储分类字符串

## 接口变化

### 新增接口

| 方法 | 路径 | 说明 |
| --- | --- | --- |
| GET | `/categories` | 查询全部启用分类列表 |
| POST | `/categories` | 新增分类 |
| PUT | `/categories/{id}/status` | 启用/禁用分类 |
| POST | `/stocks/{productId}/restock` | 补货（仓库→货架） |
| GET | `/stocks` | 返回值新增 warehouseQuantity、shelfQuantity 字段 |

### 修改接口

| 接口 | 变更说明 |
| --- | --- |
| GET `/stocks` | 响应新增 `warehouseQuantity`、`shelfQuantity`，`quantity` 字段废弃 |
| GET `/stocks/{productId}` | 同上 |
| POST `/products` | 请求新增 `categoryId`（必填）、`unit`（必填）；`category` 字段废弃 |
| GET `/products` | 响应中 `category` 改为 `categoryId` + `categoryName` |

### 鉴权规则
- 分类管理接口（增删改）：限管理员角色
- 分类查询接口：所有已登录用户可访问
- 补货接口：库管角色及以上

## 数据一致性

### 补货的原子性
补货操作必须在同一数据库事务中完成，否则可能出现"仓库库存已扣、货架库存未加"的中间状态。建议使用 `@Transactional` 注解或手动 `BEGIN/COMMIT`。

### 商品分类 FK 约束
`product.category_id` 外键约束到 `category.id`。禁用分类不会破坏已有关联（物理行仍存在），但新增商品时不允许选择 status=0 的分类（后端校验）。

### 历史 stock_log 兼容
旧有日志中 `change_type` 只有 INBOUND/OUTBOUND/CHECK，新 `stock_type` 列已在 SQL 中按规则回填，逻辑不影响历史查询。

## 后端验证

- **单元测试**：
  - `RestockService.restock(productId, N)` 仓库不足时抛出异常，不修改数据库
  - `RestockService.restock(productId, N)` 正常时仓库减 N、货架加 N、restock_order 有记录
  - `OutboundService.outbound()` 货架不足时抛出明确提示，不扣减仓库库存
- **集成测试**：
  - 入库 → 仓库库存变化，货架不变
  - 补货 → 仓库减少，货架增加，总量不变
  - 出库 → 货架减少，仓库不变
  - 盘点 → system_quantity 等于 warehouse + shelf
- **手动接口验证**：
  - Postman 测试补货接口，断言数据库中两字段均正确变化
  - 测试补货超量场景，断言返回 4xx 并含错误说明
