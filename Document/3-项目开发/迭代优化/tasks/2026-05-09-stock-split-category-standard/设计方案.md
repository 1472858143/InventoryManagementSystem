# 设计方案

## 问题定义

### 问题一：库存无法区分仓库与货架

`stock` 表当前只有 `quantity` 一个库存数量字段，无法区分商品在仓库中的数量和已经摆放在货架上可供顾客购买的数量。现实超市中：
- 顾客只能购买货架上的商品（`shelf_quantity`）
- 仓库中的商品需要由员工补货到货架上（`restock` 操作）
- 入库是供应商送货进仓库（增加 `warehouse_quantity`）

缺少这一区分导致：出库逻辑不准确（从"总库存"扣减，而非从"货架库存"），补货操作无法记录，库存状态无法反映实际业务。

### 问题二：库存单位缺失

商品数量字段没有单位，"10"可能是 10 件、10 箱或 10 kg，对称重商品、容积商品语义含糊。

### 问题三：商品分类为自由文本，无法保证一致性

`product.category` 是 `VARCHAR(50)` 自由输入，同一类商品可能被录入为"饮料"、"饮品"、"drinks"等多种写法，无法在报表和统计中合并，也无法保证前端展示的一致性。

---

## 现状分析

- **当前前端行为**：库存页显示单一"库存数量"列；商品页分类为文本输入框；无补货功能入口
- **当前后端行为**：入库 → `stock.quantity += N`；出库 → `stock.quantity -= N`；无补货操作
- **当前数据库结构**：
  - `stock(id, product_id, quantity, min_stock, max_stock, update_time)`
  - `product(id, ..., category VARCHAR(50), ...)`，无 `unit` 字段
  - 无 `category` 独立表，无 `restock_order` 表
- **当前文档描述**：正式 SQL 与以上一致，无库存分层设计

---

## 方案选择

### 子问题一：库存拆分

#### 方案 A：在 `stock` 表增加两个新字段，保留原 `quantity` 字段
- 内容：新增 `warehouse_quantity`、`shelf_quantity`，`quantity` 改为视图或保留冗余字段（`quantity = warehouse + shelf`）
- 优点：旧代码兼容性最好，改动范围小
- 风险：`quantity` 成为冗余字段，容易出现三个值不一致的状态，维护负担大；业务含义混乱

#### 方案 B：将 `stock.quantity` 替换为 `warehouse_quantity` + `shelf_quantity`，移除旧字段
- 内容：DDL 新增两列并迁移数据，待验证后删除旧 `quantity` 列；全量修改后端引用
- 优点：数据库结构干净，字段语义明确，无冗余，符合单一真相来源原则
- 风险：是破坏性变更，后端所有引用 `quantity` 的地方都要修改；需要数据迁移脚本

**推荐方案 B**：结构清晰，短期改动量大但长期维护成本低。数据迁移脚本简单（将现有 `quantity` 全部归入 `warehouse_quantity`，`shelf_quantity` 初始为 0）。通过分阶段上线可控风险。

### 子问题二：库存单位

#### 方案 A：`product` 表新增 `unit VARCHAR(20)` 自由文本，附推荐值列表
- 内容：新增字段，前端提供常用单位下拉（但允许自定义输入）
- 优点：灵活，覆盖特殊单位
- 风险：与分类同样的脏数据风险，但单位的多样性本身合理

#### 方案 B：`product` 表新增 `unit ENUM(...)` 枚举
- 内容：枚举固定常用单位
- 优点：完全约束，无脏数据
- 风险：需求变化时需要 DDL 修改，灵活性差

**推荐方案 A**：单位的多样性天然比分类高，自由文本附前端推荐值列表是合理折中。DEFAULT '件'，必填。

### 子问题三：商品分类标准化

#### 方案 A：独立 `category` 表 + 外键约束
- 内容：新增 `category(id, category_name, status, create_time)` 表；`product.category_id` FK 引用；旧 `category VARCHAR` 列迁移后删除
- 优点：分类名称只有一份真相；外键从数据库层禁止孤立分类；支持分类管理（增删改查）
- 风险：需要数据迁移；后端增加分类管理 CRUD；前端商品表单由输入框改为下拉

#### 方案 B：保留 `category VARCHAR`，但前端改为枚举下拉（后端不约束）
- 内容：不改数据库，前端限制输入范围
- 优点：改动极小
- 风险：后端/API 仍可写入任意字符串；历史脏数据不清理；无法管理分类列表

**推荐方案 A**：从数据库层保证一致性，是唯一能真正解决脏数据问题的方案。迁移脚本从现有 `category` 字符串自动抽取去重后建表。

## 推荐方案汇总

| 子问题 | 推荐 | 关键决策 |
| --- | --- | --- |
| 库存拆分 | 方案 B：替换字段 | 数据迁移：旧 quantity 全归 warehouse_quantity，shelf_quantity=0 |
| 库存单位 | 方案 A：VARCHAR + 推荐列表 | DEFAULT '件'，NOT NULL，前端提供常用值 |
| 分类标准化 | 方案 A：独立表 + FK | 从现有字符串自动抽取分类，迁移后删除旧列 |

## 库存流转模型（推荐）

```
供应商 ──[入库 inbound]──► warehouse_quantity
                                    │
                          [补货 restock]（仓库→货架，原子事务）
                                    │
                                    ▼
                            shelf_quantity ──[购买 outbound]──► 顾客

盘点（stock_check）：system_quantity = warehouse_quantity + shelf_quantity
预警（min_stock/max_stock）：应用于 total（warehouse + shelf），保持现有逻辑不变
```

## 设计决策（已由用户确认，2026-05-09）

1. **min_stock / max_stock 预警阈值** → 应用于**上架库存（shelf_quantity）**
   - 预警逻辑：`shelf_quantity < min_stock` 时标记"上架不足"；`shelf_quantity > max_stock` 时标记"上架过多"
   - 含义：提醒管理员货架快空（需补货）或货架过满（已够，无需再补货）
   - 仓库库存不参与预警判断

2. **盘点（stock_check）** → `system_quantity = warehouse_quantity + shelf_quantity`（总量）
   - 盘点员清点全部实物（不区分在库还是上架），与系统总量对比
   - 盘点调整策略：差值优先调整 `warehouse_quantity`；若仓库库存不足以吸收负差值，再从 `shelf_quantity` 扣减

3. **分类管理** → 需要独立分类管理页面（`/categories`）
   - 前端新增 `CategoryView.vue`，支持查看、新增、启用/禁用分类
   - 侧边栏菜单新增"商品分类"入口

4. **单位字段** → 必填，DEFAULT '件'（已在设计中体现）

## 影响边界

- **会修改**：`stock` 表结构、`product` 表结构、后端 Stock/Product 相关 Entity/Mapper/Service/Controller、前端库存页和商品页
- **不会修改**：用户、角色、权限、`user_role` 相关代码；现有入库/出库记录表结构（只改业务逻辑含义）；`stock_check` 表结构（`system_quantity` 语义变为总库存）
- **需要用户确认**：min_stock/max_stock 预警应用范围（上述 3 个设计问题）

## 回滚策略

由于本任务包含数据库结构性变更，回滚步骤：

1. 恢复 `stock` 表：重新加回 `quantity` 列，从 `warehouse_quantity + shelf_quantity` 计算填充，删除两个新列
2. 恢复 `product` 表：删除 `category_id` 列（FK 先 DROP）和 `unit` 列，原 `category VARCHAR` 列若已删则需从备份恢复
3. 重新部署旧版后端代码（Git 回滚至本任务开始前的提交）
4. **建议**：在数据库执行变更前做完整备份（`mysqldump`），回滚直接还原备份最稳妥
