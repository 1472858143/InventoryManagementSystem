# Agent 执行提示

## 任务目标

请先阅读本任务目录中的全部文件，再开始分析或修改。执行目标以 `README.md` 中的目标和验收标准为准。

本任务涉及数据库结构性变更，影响范围大，执行时必须严格按阶段推进：数据库变更 → 后端适配 → 前端更新。每个阶段完成后必须验证再推进下一阶段。

## 执行约束

- 不直接修改正式项目书、需求分析和系统设计文档正文。
- 不直接覆盖 `sql/market.sql` 或 `Document/3-项目开发/market.sql`。
- 数据库调整必须先写入本任务目录的 `数据库变更.sql`，用户确认后才能执行到真实数据库。
- 修改前检查 Git 状态，避免覆盖用户未提交内容。
- 每次变更保持范围小而明确，单一 PR 只包含一个阶段的变更。
- 补货操作（restock）必须在数据库事务中完成，不允许非原子的分步扣减。
- 所有库存字段的非负约束来自数据库 CHECK 约束，后端校验是第二道防线，不能只依赖后端。

## 必读文件（按顺序）

1. 根目录 `AGENTS.md`
2. `Document/3-项目开发/迭代优化/AGENT_WORKFLOW.md`
3. 当前任务 `README.md`（已读）
4. 当前任务 `设计方案.md`
5. 当前任务 `数据库变更.sql`
6. 当前任务 `后端影响.md`
7. 当前任务 `前端影响.md`
8. 现有数据库结构 `sql/market.sql`（基线参考）

## 分阶段执行建议

### 阶段一：数据库
1. 在开发数据库执行 `数据库变更.sql` 中的 DDL
2. 执行数据迁移脚本（将现有 `quantity` 归入 `warehouse_quantity`，分类字符串迁移到 `category` 表）
3. 验证所有表结构和约束，记录到 `测试记录.md`

### 阶段二：后端
1. 新增/修改实体类（Stock、Product、Category、RestockOrder）
2. 新增/修改 Mapper（含 XML 或注解 SQL）
3. 新增/修改 Service（StockService 补货事务、CategoryService CRUD）
4. 新增/修改 Controller（补货接口、分类接口）
5. 使用 Postman 或 curl 验证全部接口

### 阶段三：前端
1. 更新库存页显示三列库存和补货弹窗
2. 更新商品新增/编辑：分类改下拉、增加单位字段
3. 构建并手动验证各页面功能

## 输出要求

- 汇报实际修改文件（精确到类名或函数名）。
- 汇报验证命令和结果（命令 + 实际输出截图或文字）。
- 汇报未完成事项和风险（列明原因，不伪造通过状态）。
- 不把未验证内容表述为已完成。
