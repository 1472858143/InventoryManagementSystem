# 后端影响

## 是否涉及后端

- 结论：是，P1 与 P4 涉及后端调整。
- 影响 Controller：
  - `StockController`（新增 PATCH shelf-status / 批量改状态接口）
  - `ReportController`（新增 5 个仪表盘聚合接口）
- 影响 Service：
  - `StockService` / `StockServiceImpl`（字段适配 + 改状态方法）
  - `ReportService` / `ReportServiceImpl`（新增聚合方法）
  - `InboundService` / `OutboundService`（库存联动逻辑改为只动 quantity）
  - `StockcheckService`（盘点调整 quantity 而非双数量）
- 影响 Mapper：
  - `StockMapper` 与 `StockMapper.xml`（字段名 + 新增聚合 SQL）
  - `ReportMapper` 与 `ReportMapper.xml`（新增 5 个聚合查询）
- 影响实体或 DTO：
  - `Stock` 实体：`warehouseQuantity` → `quantity`，删除 `shelfQuantity`，新增 `shelfStatus`
  - `StockOverviewItem` VO：字段对齐
  - 新增 `RestockSuggestionItem`、`SlowMovingItem`、`SalesHotspotItem`、`CategoryDistributionItem`、`StockTurnoverItem` VO

## 业务逻辑变化

- **入库 InboundService**：保存入库单后联动 `stock.quantity += 数量`；不再写 `warehouse_quantity` 与 `shelf_quantity`。
- **出库 OutboundService**：保存出库单后联动 `stock.quantity -= 数量`；不再扣减 `shelf_quantity`，校验改为 `quantity >= 出库数量`。
- **盘点 StockcheckService**：调整 `quantity` 而非双数量；shelf_status 不变。
- **新增上架补货逻辑 StockService.updateShelfStatus(id, status)**：
  - 校验 status 在 `{未上架, 缺货, 较少, 充足}` 内。
  - 更新单条 stock 的 shelf_status。
  - 写入 stock_log，stock_type='shelf_status_change'，记录 from/to 状态。
- **新增批量上架补货 StockService.batchUpdateShelfStatus(ids, status)**：
  - 校验同上。
  - 事务中批量更新。
  - 每条单独写日志。
- **新增 ReportService 聚合方法**：
  - `getStockTurnover(startDate, endDate)`：库存周转率 = 期间出库总量 / 期间平均库存。
  - `getRestockSuggestions()`：返回 quantity < min_stock 或 shelf_status='缺货' 的商品列表，含建议补货数量。
  - `getSlowMoving(days)`：返回最近 N 天无出库的商品列表。
  - `getSalesHotspot(days)`：按出库总量排序的 Top N 商品。
  - `getCategoryDistribution()`：按分类聚合的库存数量与价值占比。

## 数据一致性

- **入库/出库与 stock.quantity 的事务**：当前 InboundService/OutboundService 通过 `@Transactional` 保证库存联动。本任务保持事务边界，仅调整字段名。
- **shelf_status 与 quantity 无强一致性约束**：理货员手动维护 shelf_status，系统不强制 "quantity=0 时 shelf_status 必须=缺货"——这是设计选择（贴近真实运营，理货员判断优先）。仪表盘可在数据不一致时给出提示（如"商品 X 的 quantity=0 但 shelf_status=充足，请核对"）。
- **stock_log 记录粒度**：上架补货操作每次状态变更都写日志，便于审计。
- **聚合接口性能**：库存周转率涉及历史出库聚合，若 outbound_order 数据量大需考虑加索引（current `outbound_order.create_time` 已有索引基础）。
