# 设计方案

## 问题定义

毕业设计超市库存管理系统当前前端存在 6 类问题，影响演示质感和业务故事完整度：

1. **信息架构混乱**：侧边栏 10 项菜单平铺无分组，业务流程相关项（商品/分类/入库/出库/盘点/库存）没有归类，"用户管理"作为系统功能却出现在第二位。
2. **业务表达缺失**：页面以表格 + 表单为主，缺少超市运营场景化表达（缺货预警、补货提示、滞销识别、决策辅助）。
3. **模块边界不清**：商品管理、入库、出库、盘点散落，缺少统一的业务工作台串联。
4. **数据模型与运营脱节**：IT-001 的 `shelf_quantity` 精确数量字段与真实超市运营不符——理货员只关注陈列状态（缺货/较少/充足），不追踪货架实时数量。
5. **报表深度不足**：现有 3 个图表无法体现多维度商业分析，缺少决策辅助。
6. **视觉系统过时**：沿用 Element Plus 默认配色 + ant-design 风格图标，缺少现代感。

## 现状分析

- **当前前端行为**：
  - `frontend/src/router/index.js` 定义 10 个平铺路由 + 1 个登录页。
  - `frontend/src/layout/AdminLayout.vue` 用 Element Plus `el-menu` 实现单层 sidebar，深蓝 `#001529` + ant-design 图标。
  - 各业务页面 `views/{product,category,stock,inbound,outbound,stockcheck,report,system,user}/View.vue` 均为独立 CRUD 表格 + 表单 + 详情对话框。
  - HomeView 提供 4 张统计卡 + 2 张 7 天趋势图；ReportView 提供 3 张图表（库存总览、入库 30 天趋势、出库 30 天趋势）。
- **当前后端行为**：
  - Stock entity 字段为 `warehouseQuantity` + `shelfQuantity`（IT-001 后），`product` 表已有 `category_id` 与 `unit`。
  - Report 模块仅提供 3 个聚合接口：`/reports/stock-overview`、`/reports/inbound-trend`、`/reports/outbound-trend`。
- **当前数据库结构**（IT-001 后）：
  - `stock` 表：`product_id`, `warehouse_quantity`, `shelf_quantity`, `min_stock`, `max_stock`，旧 `quantity` 字段保留待删除。
  - `stock_log` 表：`stock_type` 字段已新增。
- **当前文档描述**：
  - 系统设计文档与项目书未涉及"陈列状态"概念。
  - API 设计文档已同步至 IT-001 后状态（system + report 模块）。

## 方案选择

### 方案 A：保留 IT-001 双数量结构，仅前端重设计

- 内容：不动数据库，前端把 `shelf_quantity` 转换为状态徽章展示（如 0=缺货，<min=较少，>=min=充足）。
- 优点：不需要数据库迁移，IT-001 投入不浪费。
- 风险：前端展示与底层数据语义错位（数据库说"5 个货架"，前端说"较少"，但实际货架可能空也可能没空）。理货员补货后必须录入精确数字，违背用户提出的"陈列状态而非精确数量"判断。中长期累积数据失真，影响仪表盘准确性。

### 方案 B：修正数据模型为"总库存数量 + 陈列状态枚举"

- 内容：
  - `stock.warehouse_quantity` 重命名为 `stock.quantity`（总库存数量）。
  - 删除 `stock.shelf_quantity`。
  - 新增 `stock.shelf_status` enum：`未上架` / `缺货` / `较少` / `充足`，完全手动维护。
  - 入库 → `quantity +N`，出库 → `quantity -N`，上架补货 → 改 `shelf_status`。
- 优点：
  - 数据模型贴近真实超市运营（理货员手动标记陈列状态）。
  - 业务动作语义清晰：入库出库管数量，上架补货管状态。
  - 仪表盘聚合基于精确总数量，分析维度可靠。
- 风险：
  - 需要 schema 迁移（数据迁移脚本 + 后端字段适配）。
  - 部分回退 IT-001 投入（`shelf_quantity` 删除）。

### 方案 C：双轨保留，shelf_status 与 shelf_quantity 共存

- 内容：保留 `shelf_quantity` 不变，新增 `shelf_status`，由用户选择哪个生效。
- 优点：兼容性最好，IT-001 完全保留。
- 风险：两个字段语义重叠，业务规则复杂；前端展示决策困难；数据一致性维护成本高；未来报表逻辑膨胀。

## 推荐方案

**采用方案 B**。

理由：
- 与用户经过论证后提出的业务模型一致（"货架精确数量不合理，应该用陈列状态描述"）。
- IT-001 的双层流转思想被保留（双层确实存在），只是底层度量从"精确数量"调整为"数量 + 状态"，更准确。
- 业务动作语义清晰，便于演示讲故事（入库管数量、上架管状态）。
- 仪表盘聚合的数据来源单一（`quantity`），可靠性高。

后续整体设计基于方案 B 推进，详细决策见下文。

## 核心设计决策

### 决策 1：信息架构（首页驾驶舱 + 4 个业务分组）

```
🏠 首页（驾驶舱）
🛍 商品管理
  ├─ 商品库（CRUD + 上下架状态）
  └─ 商品分类
📦 库存管理
  ├─ 工作台（缺货预警 + 陈列状态分布 + 滞销提示 + 任务入口）
  ├─ 库存总览（quantity + shelf_status 双层展示）
  ├─ 入库单据（quantity +N）
  ├─ 上架补货（改 shelf_status）
  ├─ 出库单据（quantity -N）
  └─ 库存盘点
📈 报表分析（多 Tab 商业智能仪表盘）
⚙ 系统
  ├─ 用户管理
  └─ 系统信息
```

分组原则：商品管理 = 静态主数据（"卖什么"），库存管理 = 动态状态 + 流转动作（"现在有多少、在哪、怎么动"）。

### 决策 2：模块整合形态（B+C 模式）

- **B 二级侧边菜单**：一级菜单展开二级子菜单，URL 直达，与现有 Element Plus 模式兼容。
- **C 工作台首屏**：库存管理一级入口默认进入工作台首屏（指标卡 + 任务入口 + 预警），点击具体任务再跳转 CRUD 页。
- 商品管理无独立工作台（直接进商品库，避免空泛指标卡）。

### 决策 3：视觉系统

- **主色**：`#6366F1`（indigo-500）
- **背景**：`#F8FAFC`（slate-50）
- **侧栏**：`#FAFBFF`
- **文字主色**：`#0F172A`（slate-900）
- **文字辅助**：`#64748B`（slate-500）
- **边框**：`#EEF0F5`
- **状态色**：成功 `#10B981`（emerald-500） / 警告 `#F59E0B`（amber-500） / 危险 `#EF4444`（red-500）
- **图标系统**：Lucide Icons（线性、统一描边粗细 1.5px）替代当前 ant-design 风格图标。
- **设计令牌**：集中在 `frontend/src/styles/tokens.css`，定义 CSS 变量供全局使用。
- **组件圆角**：8-10px；卡片阴影 `0 1px 3px rgba(15, 23, 42, 0.04)`。

### 决策 4：数据模型修正（IT-001 部分回退 + 新增）

| 字段 | 当前（IT-001 后） | 调整后 | 迁移逻辑 |
| --- | --- | --- | --- |
| `stock.warehouse_quantity` | int | 重命名为 `stock.quantity` | 直接 RENAME |
| `stock.shelf_quantity` | int | 删除 | 数据合并到 quantity（迁移时 quantity = warehouse_quantity + shelf_quantity） |
| `stock.shelf_status` | 不存在 | 新增 enum | 默认值 `未上架`；初始迁移时按 quantity 推算（=0 未上架，>0 充足，由用户上线后再手动调整） |

**业务动作语义**：
- 入库单 → `quantity += 数量`；不影响 `shelf_status`。
- 出库单 → `quantity -= 数量`；不影响 `shelf_status`（理货员需巡视后手动改）。
- 上架补货 → 改 `shelf_status`（如从"缺货"改"充足"）；不影响 `quantity`（仍在仓库总量内）。
- 库存盘点 → 调整 `quantity`，不影响 `shelf_status`。

### 决策 5：核心场景

**场景 1：库存数量 + 陈列状态双层流转**

```
[供应商送货] → 入库单据登记 (quantity +50)
              → shelf_status 仍为"未上架"
[理货员上架] → 上架补货页面，将 shelf_status 改为"充足"
[顾客购买]   → 出库单据登记 (quantity -10)
              → shelf_status 不变（仍"充足"）
[巡视发现空]  → 上架补货页面，将 shelf_status 改为"缺货"
              → 库存工作台显示"3 个商品缺货"预警
[再次补货]   → 上架补货页面，将 shelf_status 改回"充足"
```

涉及页面：库存工作台、库存总览、入库单据、上架补货、出库单据、库存盘点。

**场景 2：商业智能仪表盘**

报表分析重设计为多 Tab 仪表盘：

- **Tab 1 库存健康度**：库存周转率、库存账龄分布、库存价值、各分类库存占比饼图。
- **Tab 2 销售分析**：销售热点商品 Top10、各分类销售占比、近 30 天销售趋势对比。
- **Tab 3 决策辅助**：补货建议清单（基于出库速度 + 当前 quantity）、滞销商品识别（30 天无出库）、缺货商品清单（shelf_status='缺货'）。
- **Tab 4 流转明细**：入库出库趋势对比、入库出库 Top 商品。

涉及页面：报表分析（重设计为多 Tab 仪表盘）、对应后端聚合接口。

### 决策 6：阶段划分（场景纵切法，5 阶段）

| Phase | 范围 | 完成标志 |
| --- | --- | --- |
| **P1 数据模型迁移** | 数据库 schema 变更 + 后端 entity/mapper/VO/Service 适配 + 前端 API 类型对齐 | `mvn test` 通过；前端原页面在新 API 下可读可写 |
| **P2 IA + 视觉基线** | router 重构、AdminLayout 分组侧边栏、Lucide 图标接入、tokens.css 设计令牌、共享组件 PageContainer/StatCard 抽取、原 10 个页面在新骨架上跑通 | 全站新视觉，但页面内部仍是简单 CRUD |
| **P3 场景一：双层流转** | 库存工作台（新页面）、库存总览重设计（双层展示）、入库/上架补货/出库/盘点流程化 | 演示完整双层流转故事 |
| **P4 场景二：BI 仪表盘** | 报表分析重设计为多 Tab 仪表盘 + 后端聚合接口扩展 | 演示完整决策辅助故事 |
| **P5 首页驾驶舱 + 收尾** | 首页驾驶舱重设计（聚合两大场景指标）、商品管理细节、系统模块对齐、整体走查 | 整体一致性达标 |

每个 Phase 完成后产出一个可独立演示的中间状态，并提交一次 Git commit。

## 影响边界

- **会修改**：
  - 数据库 `stock` 表 schema（删除 shelf_quantity、重命名 warehouse_quantity → quantity、新增 shelf_status）。
  - 后端 `Stock` 实体、Mapper、Service、Controller、StockMapper.xml。
  - 后端 Report 模块新增聚合接口（库存周转率、补货建议、滞销识别等）。
  - 前端 router、layout、所有 view 组件、styles。
  - 前端新增 `StockWorkspaceView`、`ShelfRestockView`、报表多 Tab 子页。
- **不会修改**：
  - 数据库 `product`、`category`、`user`、`inbound_order`、`outbound_order`、`stock_log` 表结构。
  - 后端 `auth`、`user`、`product`、`category`、`inbound`、`outbound`、`stockcheck`、`system` 模块业务逻辑。
  - 正式 SQL（`sql/market.sql`、`Document/3-项目开发/market.sql`）—— 验收后再考虑同步。
  - 项目书、需求分析、系统设计文档正文 —— 验收后再考虑同步。
- **需要用户确认**：
  - P1 完成后，老 `quantity` 字段（IT-001 遗留待删）的处理方式。
  - P3 完成后，缺货预警的阈值规则（基于 shelf_status 还是 min_stock）。
  - P4 完成后，仪表盘指标公式细节（如库存周转率分母选择）。

## 回滚策略

- **P1 失败回滚**：保留 schema 迁移脚本的 DOWN 方向（恢复 `shelf_quantity` + 撤销 `quantity` 重命名 + 删除 `shelf_status`），后端代码 git revert 到 P1 之前。
- **P2 失败回滚**：前端 layout/router 改动 git revert，原页面文件不动，可立即恢复 IT-001 后状态。
- **P3-P5 失败回滚**：各 Phase 独立 commit，git revert 对应 commit 即可恢复至上一 Phase 完成态。
- **整体放弃回滚**：保留 IT-001 验收后的 git commit hash，必要时 `git reset --hard` 回到该 commit（仅在用户确认后执行）。
