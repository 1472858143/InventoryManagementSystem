# 前端影响

## 是否涉及前端

- 结论：涉及。
- 影响页面：新增 `frontend/src/views/purchase/` 与 `frontend/src/views/sales/` 下 8 个页面；调整菜单与路由。
- 影响组件：单据明细行编辑、商品选择、供应商 / 客户选择、金额汇总、作废确认、查询筛选与分页组件。
- 影响 API 封装：新增 `frontend/src/api/purchase.js`、`frontend/src/api/purchaseReturn.js`、`frontend/src/api/sales.js`、`frontend/src/api/salesReturn.js` 或等价封装。

## 用户交互变化

新增“进货管理”菜单组：

- 进货入库：选择供应商，添加多商品明细，保存后库存增加。
- 采购退货：选择供应商和可选原进货单，添加退货明细，保存后库存减少。
- 进货单查询：按时间段 / 供应商 / 单据号查询，支持查看明细与作废。
- 采购退货查询：同上结构。

新增“销售管理”菜单组：

- 销售出库：选择客户，添加多商品明细，保存后库存减少。
- 客户退货：选择客户和可选原销售单，添加退货明细，保存后库存增加。
- 销售单查询：按时间段 / 客户 / 单据号查询，支持查看明细与作废。
- 客户退货查询：同上结构。

## 数据契约变化

新增 API 契约预计包括：

- 进货：创建、查询列表、查询详情、作废。
- 采购退货：创建、查询列表、查询详情、作废。
- 销售：创建、查询列表、查询详情、作废。
- 客户退货：创建、查询列表、查询详情、作废。

前端请求需提交主表信息和明细数组；响应需包含单据状态、总数量、总金额、明细、对方主体名称、操作人和创建时间。

## 兼容性风险

- 8 个页面体量较大，路由与菜单变更可能影响现有布局。
- 明细表单需要稳定处理商品重复、数量为 0、库存不足、金额精度等错误提示。
- 旧“入库 / 出库”菜单本任务不下线，避免影响旧演示流程；新菜单不应调用旧接口。
- 前端金额仅作为展示和提交参考，最终小计、总数量和总金额以后端返回为准。
- 作废按钮必须以后端状态为准；已作废单据禁用作废入口，重复作废仍由后端兜底。

## 前端验证

- 构建命令：`cd frontend && npm run build`。
- 手动验证路径：4 类单据新增、查询、详情、作废；库存数量变化；负库存提示；筛选分页。
- 自动化测试：暂无既有前端测试框架，优先补手动验证记录；如新增前端测试能力，需覆盖明细行和金额汇总。
