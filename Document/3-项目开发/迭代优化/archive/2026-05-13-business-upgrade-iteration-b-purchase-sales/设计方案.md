# 设计方案

## 设计结论

IT-006 按完整“迭代 B：进货 + 销售 + StockChangeService”设计，覆盖进货入库、采购退货、销售出库、客户退货、8 张业务单据表、`stock_log` 扩展、统一库存变更入口和 8 个前端页面。

实施时允许拆为 B1 / B2 分阶段交付，但第一阶段必须先落地 `StockChangeService`。后续所有新业务库存变化必须通过该入口，不允许业务单据 Service 直接更新 `stock.quantity`。

## 问题定义

当前系统仍以旧的单商品入库 / 出库流程承载库存变化，无法表达供应商进货、采购退货、客户销售、客户退货的完整进销存业务链路，也缺少统一的库存变更入口和可追溯来源字段。IT-006 需要把进货与销售业务升级为多商品单据模型，并让所有新库存变化通过 `StockChangeService` 统一处理。

## 现状分析

- 当前前端行为：已有旧“入库”“出库”菜单和页面；IT-005 已新增基础资料菜单组，供应商 / 客户主数据可作为进货 / 销售单据前置数据。
- 当前后端行为：旧 `InboundService` / `OutboundService` 可写库存，但不是本次新业务的目标入口；库存增减逻辑分散，缺少统一来源追踪。
- 当前数据库结构：已有 `supplier`、`customer`、`stock`、`stock_log`；`stock_log` 仍缺 `source_type`、`source_id`、`reason`、`operator_id` 等新业务追踪字段。
- 当前文档描述：IT-004 已确定采用主表 + 明细表、保存即生效、作废反向冲销、行锁防并发负库存。

## 方案选择

### 方案 A：按业务链路垂直实现

- 内容：先做进货，再采购退货，再销售，再客户退货。
- 优点：每条业务链路可以独立验收。
- 风险：库存逻辑容易在不同 Service 中重复，若先做业务再抽核心，后续统一库存入口风险更高。

### 方案 B：库存核心优先 + 业务链路分阶段落地（推荐）

- 内容：先落地 `StockChangeService`、`stock_log` 来源字段、通用状态和作废规则；再按 B1 进货、B2 销售落地四类单据。
- 优点：库存一致性先收口，四类单据共享同一变更规则；后续 IT-007 / IT-008 可直接复用。
- 风险：第一阶段必须把库存核心测试补扎实，否则后续业务都会受影响。

### 方案 C：抽象通用单据引擎

- 内容：把采购、销售、退货抽象为通用 OrderService / OrderItem 模型。
- 优点：复用最多。
- 风险：当前项目规模不需要这么重，容易过度设计，调试和验收成本高。

## 推荐方案

采用方案 B。文档边界覆盖完整 IT-006，开发执行可拆分阶段：

1. 数据库 + `StockChangeService`。
2. 进货入库 + 采购退货。
3. 销售出库 + 客户退货。
4. 查询页、作废、回归和文档收口。

## 总体架构

IT-006 采用“统一库存核心 + 四类业务单据”的结构。

```text
PurchaseService / PurchaseReturnService / SalesService / SalesReturnService
        -> StockChangeService.apply(...)
              -> 锁定 stock 行
              -> 校验库存
              -> 更新 stock.quantity
              -> 写入 stock_log(source_type, source_id, reason, operator_id)
```

- `stock` 模块新增 `StockChangeService`，作为新业务唯一库存变更入口。
- `purchase` 模块负责进货入库、采购退货。
- `sales` 模块负责销售出库、客户退货。
- `stock_log` 作为所有后续统计报表的数据来源，新增记录必须带 `source_type` 和 `source_id`。
- 旧 `InboundService` / `OutboundService` 不改、不删、不迁移，保持旧功能可用；新菜单不再调用旧接口。

### 约束取舍

- 单据编号设置唯一索引。
- 对方主体、商品、库存存在性在代码层校验。
- 本轮不新增外键到正式 SQL，减少迭代 B 体量和迁移阻塞；索引与唯一约束必须保留。
- `stock_log` 新字段允许 `NULL`，兼容旧日志；但 IT-006 新写入记录必须带 `source_type` 与 `source_id`。
- `subtotal`、`total_quantity`、`total_amount` 由后端计算，前端提交值不可信。

## 作废规则

单据状态只允许：

```text
NORMAL -> CANCELLED
```

不支持恢复、不支持物理删除。作废时：

- 主表状态改为 `CANCELLED`。
- 原明细不改。
- 原 `stock_log` 不改。
- 按原明细逐条写反向库存日志。
- 反向日志 `reason` 固定包含“作废冲销：{单据号}”。
- 作废操作在同一个事务里完成。

方向示例：

```text
进货入库 +10，作废时 -10
采购退货 -5，作废时 +5
销售出库 -3，作废时 +3
客户退货 +2，作废时 -2
```

作废失败规则：

- 单据不存在：404。
- 已作废：400，“单据已作废”。
- 反向冲销导致库存不足：400。例如客户退货作废需要扣库存，当前库存不足则阻止作废。

## StockChangeService 设计

### 接口

```java
public interface StockChangeService {
    StockChangeResult apply(StockChangeCommand command);
}
```

`StockChangeCommand`：

```java
public record StockChangeCommand(
    Long productId,
    Integer delta,
    SourceType sourceType,
    Long sourceId,
    Long operatorId,
    String reason
) {}
```

`StockChangeResult`：

```java
public record StockChangeResult(
    Long productId,
    Integer beforeQuantity,
    Integer afterQuantity
) {}
```

`SourceType` 枚举：

```java
PURCHASE_IN,
PURCHASE_RETURN_OUT,
SALE_OUT,
SALE_RETURN_IN,
STOCK_LOSS,
STOCK_OVERFLOW,
STOCK_CHECK
```

### 核心规则

`delta` 使用统一语义：

```text
正数：库存增加
负数：库存减少
0：不允许，直接报错
```

`apply()` 内部步骤：

1. 校验 `productId`、`delta`、`sourceType`、`sourceId` 非空。
2. 使用 `StockMapper.findEntityByProductIdForUpdate(productId)` 锁定库存行。
3. 若库存记录不存在，返回业务异常“库存记录不存在”。
4. 计算 `after = before + delta`。
5. 若 `after < 0`，返回业务异常“库存不足，当前库存：{before}”。
6. 更新 `stock.quantity = after`。
7. 写入 `stock_log`：
   - `change_type` 保留兼容：正数写 `INBOUND`，负数写 `OUTBOUND`。
   - `stock_type` 固定写 `WAREHOUSE`。
   - `change_quantity` 写实际 `delta`。
   - `before_quantity` / `after_quantity` 写锁定后的库存前后值。
   - `source_type` / `source_id` / `reason` / `operator_id` 写业务来源。
8. 返回变更前后库存。

### 事务与并发

`StockChangeService.apply()` 自身加 `@Transactional`，但主要作为业务单据事务的一部分运行。四类单据 Service 的创建 / 作废方法也加 `@Transactional`：

```text
创建单据主表
插入明细
逐条 apply 库存变化
任意一步失败 -> 整张单据回滚
```

新增 Mapper 方法：

```java
Stock findEntityByProductIdForUpdate(@Param("productId") Long productId);
```

SQL：

```sql
SELECT ...
FROM stock
WHERE product_id = #{productId}
FOR UPDATE
```

这样销售出库、采购退货等扣减库存的场景会串行处理同一商品库存，避免两个并发请求都看到同一个旧库存后扣成负数。

### 与旧 StockService 的关系

旧 `StockService.increaseStock/decreaseStock/adjustStock` 暂不改造，避免影响旧入库、出库、盘点页面。IT-006 新业务全部调用 `StockChangeService`。后续 IT-009 下线旧菜单后，再评估是否让旧服务委托到 `StockChangeService` 或删除旧入口。

### 后端校验

创建单据时：

- 对方主体必须存在且 `status=1`。
- 商品必须存在且 `status=1`。
- `items` 不能为空。
- `quantity > 0`。
- `unitPrice >= 0`。
- 同一张单据内不允许重复 `productId`。
- `subtotal = quantity * unitPrice` 后端计算。
- `totalQuantity`、`totalAmount` 由后端汇总。
- 扣减库存类单据库存不足时整单失败。

### 单据编号

后端生成，不由前端输入：

```text
PO  + yyyyMMddHHmmss + 4位随机数
PRO + yyyyMMddHHmmss + 4位随机数
SO  + yyyyMMddHHmmss + 4位随机数
SRO + yyyyMMddHHmmss + 4位随机数
```

唯一索引兜底，冲突时重试 2 次；仍失败返回业务异常。
