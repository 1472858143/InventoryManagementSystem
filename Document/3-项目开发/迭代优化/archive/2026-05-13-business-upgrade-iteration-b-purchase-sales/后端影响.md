# 后端影响

## 是否涉及后端

- 结论：涉及。
- 影响 Controller：新增 PurchaseController、PurchaseReturnController、SalesController、SalesReturnController 或等价拆分。
- 影响 Service：新增 `StockChangeService`、PurchaseService、PurchaseReturnService、SalesService、SalesReturnService。
- 影响 Mapper：新增 4 类单据主表 / 明细表 Mapper；扩展 StockMapper / StockLogMapper 支持行锁和来源字段。
- 影响实体或 DTO：新增单据 Entity、Item Entity、CreateRequest、QueryRequest、DetailVO、ListItemVO、CancelRequest、StockChangeCommand、SourceType。

## 业务逻辑变化

- 4 类单据保存为单事务：插入主表、插入多条明细、逐条调用 `StockChangeService`。
- 保存即生效：进货入库增加库存，采购退货减少库存，销售出库减少库存，客户退货增加库存。
- 作废采用反向冲销：单据状态从 `NORMAL` 改为 `CANCELLED`，写入取反方向的 `stock_log`，不删除原日志。
- `StockChangeService` 是新业务库存变更唯一入口，负责校验商品 / 库存、行锁、负库存拦截、更新库存、写入 `stock_log`。

## 接口变化

预计新增接口：

- `/api/purchase/orders`：创建进货单、查询列表、查询详情、作废。
- `/api/purchase/returns`：创建采购退货单、查询列表、查询详情、作废。
- `/api/sales/orders`：创建销售单、查询列表、查询详情、作废。
- `/api/sales/returns`：创建客户退货单、查询列表、查询详情、作废。

接口鉴权沿用现有登录态机制；菜单级权限拦截留给 IT-009。

## 数据一致性

- 关键风险在库存并发：`StockChangeService` 必须通过 `SELECT ... FOR UPDATE` 或等价行锁锁定 `stock` 行。
- 多商品明细中任一商品库存不足或写入失败，整张单据必须回滚。
- `stock_log.source_type` / `source_id` 是后续统计报表口径基础，新增记录不得为空。
- 旧 InboundService / OutboundService 不在本任务退役，避免旧流程回归风险。
- 作废只允许 `NORMAL -> CANCELLED`；作废时不删除原日志，必须逐条写入反向冲销日志。
- `subtotal`、`totalQuantity`、`totalAmount` 必须由后端计算，不能信任前端提交值。
- 新业务 `change_type` 保留兼容：库存增加写 `INBOUND`，库存减少写 `OUTBOUND`；`stock_type` 固定写 `WAREHOUSE`。

## 后端验证

- 单元测试：`StockChangeService` 正向 / 负向变更、负库存拦截、日志字段写入、作废取反命令。
- 集成测试：4 类单据创建、详情、查询、作废；多明细事务回滚；并发或行锁关键路径。
- 手动接口验证：通过前端或接口工具跑通进货、退货、销售、客户退货全链路。
- SQL 验证：`StockMapper.findEntityByProductIdForUpdate` 必须包含 `FOR UPDATE`；列表查询必须覆盖时间段 / 对方主体 / 单据号 / 分页。
