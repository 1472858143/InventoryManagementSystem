# Agent 执行提示

## 任务目标

请先阅读本任务目录中的全部文件，再开始分析或修改。执行目标以 `README.md` 中的目标和验收标准为准。

本任务为前端整体重设计 + 数据模型修正，按 5 个 Phase 顺序推进。每个 Phase 完成后须产出可演示的中间状态。

## 执行约束

- 不直接修改正式项目书、需求分析和系统设计文档正文。
- 不直接覆盖 `sql/market.sql` 或 `Document/3-项目开发/market.sql`。
- 数据库调整先写入本任务目录的 `数据库变更.sql`，验收后再考虑同步正式 SQL。
- 修改前检查 Git 状态，避免覆盖用户未提交内容。
- 每次变更保持范围小而明确，按 Phase 分阶段提交。
- 不跳跃 Phase：必须 P1 完成（数据库 + 后端适配）后才能开始 P2，依此类推。
- 不修改超出当前 Phase 范围的页面，避免污染其他 Phase 的可演示性。

## 必读文件

1. 根目录 `AGENTS.md`
2. `Document/3-项目开发/迭代优化/AGENT_WORKFLOW.md`
3. 当前任务 `README.md`
4. 当前任务 `设计方案.md`
5. 当前任务 `前端影响.md` 与 `后端影响.md`
6. 当前任务 `数据库变更.sql`

## Phase 边界

- **P1 数据模型迁移**：仅动 schema、entity、mapper、VO、API 类型，不动前端页面内容。允许小幅修改前端 stock.js / report.js 类型对齐，不允许重设计页面。
- **P2 IA + 视觉基线**：仅动 router、layout、共享组件、设计令牌、Lucide 图标接入。所有原页面"挂在新骨架上跑通即可"，不重设计页面内部内容。
- **P3 场景一**：仅动库存管理域的页面（库存工作台、库存总览、入库单据、上架补货、出库单据、库存盘点）。
- **P4 场景二**：仅动报表分析域 + 后端报表聚合接口。
- **P5 收尾**：首页驾驶舱、商品管理细节、系统模块视觉对齐、整体走查。

## 输出要求

- 每个 Phase 完成后：
  - 在 `开发记录.md` 时间线中记录。
  - 在 `测试记录.md` 中记录验证命令和结果。
  - 提交一次 Git commit，commit message 标注 Phase 编号（如 `feat(IT-002 P1): 数据模型迁移`）。
- 汇报实际修改文件清单。
- 汇报验证命令和结果。
- 汇报未完成事项和风险。
- 不把未验证内容表述为已完成。

## 风险提示

- IT-001 已实施 `warehouse_quantity` + `shelf_quantity` 字段拆分，本任务 P1 须回退 `shelf_quantity` 并重命名 `warehouse_quantity` → `quantity`。执行前确认 IT-001 的"待删除老字段 quantity"是否已实际删除，避免字段冲突。
- 视觉令牌建议集中在 `tokens.css`，避免散落在各组件 scoped style，否则后续主题切换无法统一。
- 仪表盘新接口若改动大，可在 P4 内分小步：先扩报表 VO，再加聚合方法，最后接前端。
