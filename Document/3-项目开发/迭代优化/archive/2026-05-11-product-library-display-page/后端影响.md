# 后端影响

## 是否涉及后端

- 结论：涉及，但改动较小且不新增对外接口。
- 影响 Controller：无（路径与签名不变）。
- 影响 Service：`product/service/impl/ProductServiceImpl.java`。
- 影响 Mapper：当前仓库为注解式 `product/mapper/ProductMapper.java`，调整其中 `findAllWithCategory` 的 SELECT 与 JOIN。
- 影响实体或 DTO：
  - `product/dto/ProductCreateRequest.java`：productCode 字段语义改为可选。
  - `product/vo/ProductListItemResponse.java`：新增 `salesCount` 字段。
  - `product/model/ProductView.java`：新增 `salesCount` 字段。

## 业务逻辑变化

### 商品编号自动生成

- **触发条件**：`createProduct` 接收到 `request.productCode()` 为 null 或 blank。
- **生成规则**：`P` + `yyyyMMddHHmmss` + 4 位随机数字（共 19 字符）。
- **碰撞处理**：依赖 `uk_product_code` 唯一索引兜底；插入抛 `DuplicateKeyException` 时重试一次（重新生成编号），仍失败则抛 `BusinessException(500, "商品编号生成失败，请重试")`。
- **校验调整**：`validateCreateRequest` 中移除「商品编码不能为空」分支；其他校验（名称、价格）保留。
- **向下兼容**：若调用方仍传入 productCode，依旧尊重传入值，逻辑等同旧实现（含「商品编码已存在」校验）。

### 商品列表附带销量

- **数据来源**：`stock_log.change_type = 'OUTBOUND'` 的 `change_quantity` 累计。
- **聚合方式**：在 `findAllWithCategory` 加 LEFT JOIN 子查询，避免 N+1。
- **缺省值**：无出库记录的商品 `salesCount` 为 0（COALESCE 兜底）。
- **不引入缓存**：聚合开销在毕设规模下可忽略；如未来商品数据量上升再评估。

## 接口变化

| 接口 | 路径与方法 | 变更 |
| --- | --- | --- |
| 商品列表 | `GET /api/products` | 响应每项新增 `salesCount: number` 字段 |
| 创建商品 | `POST /api/products` | 请求 body 中 `productCode` 字段允许缺省；缺省时由后端生成 |
| 状态切换 | `PUT /api/products/status` | 不变 |
| 启用分类 | `GET /api/categories/enabled` | 不变 |

## 数据一致性

- 不引入新的写入路径；商品编号自动生成在事务内完成，与 `stockService.initializeStockForProduct` 同事务。
- 销量聚合是只读派生数据，与 stock_log 实时一致；不会因为派生与持久化字段双写而产生不一致风险。
- `outbound_order` 与 `stock_log` 的关系：IT-002 后所有出库已通过 stock_log 统一记录（含 change_type=OUTBOUND）。本任务统一以 stock_log 为聚合源，避免 outbound_order 与 stock_log 之间的数据口径分歧。

## 后端验证

- 单元测试新增 / 调整：
  - `ProductServiceImplTest`（新增或扩充）：
    - `createProduct_whenProductCodeIsNull_shouldAutoGenerate`：传入 productCode=null，断言返回的 product 有非空 productCode。
    - `createProduct_whenProductCodeProvided_shouldKeepIt`：传入 productCode="P_TEST_001"，断言保留传入值。
    - `listProducts_shouldIncludeSalesCount`：mock mapper 返回带 salesCount 的视图，断言 VO 字段透传。
- 集成测试：依赖现有 BackendApplicationTests 启动验证；不新增端到端用例。
- 手动接口验证：
  - `POST /api/products` 不传 productCode，期望 200 + 商品列表查询能看到自动编号。
  - `GET /api/products` 响应每项含 `salesCount` 字段。
- 验证命令：`cd backend && mvn test`。

## 实施结果

- `ProductCreateRequest.productCode` 已保留并允许为空。
- `ProductServiceImpl.createProduct` 已在缺省编号时生成 `P` + `yyyyMMddHHmmss` + 4 位随机数字，并在 `DuplicateKeyException` 时重试一次。
- `ProductView` 与 `ProductListItemResponse` 已新增 `salesCount`，列表响应对空值兜底为 0。
- `ProductMapper.java` 的 `findAllWithCategory` 已加入基于 `stock_log(change_type='OUTBOUND')` 的销量聚合子查询。
- 已新增 `ProductServiceImplTest`，覆盖缺省编号、编号冲突重试、保留传入编号和销量字段透传。

## 风险与回滚

- 编号自动生成格式如未来需要业务可读性更强（例如分类前缀），可在生成器内集中调整，对外契约保持不变。
- 销量聚合若未来切到独立 sales 模块，可将 LEFT JOIN 子查询替换为新 sales 模块的视图或聚合表，VO 层无需变更。
- 回滚路径：恢复「商品编码不能为空」校验、移除 ProductMapper.xml 的 LEFT JOIN、从 VO 与 model 中移除 salesCount 字段。
