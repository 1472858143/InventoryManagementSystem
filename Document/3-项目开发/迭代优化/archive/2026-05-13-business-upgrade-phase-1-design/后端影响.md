# 后端影响

## 是否涉及后端

- 结论：涉及，但第一阶段只产出设计，不改后端代码。
- 影响 Controller：后续将新增 SupplierController、CustomerController、PurchaseController、PurchaseReturnController、SalesController、SalesReturnController、StockAdjustmentController、StockAlarmController（查询）、SystemLogController、RoleController、PermissionController、PasswordController。
- 影响 Service：新增上述对应 Service，并新增一个核心 `StockChangeService`（库存统一变更单点入口）。
- 影响 Mapper：新增 supplier / customer / purchase_order / purchase_order_item / purchase_return_order / purchase_return_order_item / sales_order / sales_order_item / sales_return_order / sales_return_order_item / stock_adjustment_order / stock_adjustment_order_item / system_log / permission / role_permission 等 Mapper。
- 影响实体或 DTO：每张新表对应 Entity；单据保存接口需要 Request DTO（主表 + 明细数组）与 Response VO；报表查询接口各自有专属 VO。
- 既有 `StockService.increaseStock / decreaseStock / adjustStock` 下沉为 `StockChangeService` 的内部细节，对外不再暴露。
- 既有 `InboundService` / `OutboundService` 在迭代 B 完成后不再被新菜单调用；迭代 E 下线旧菜单。

## 业务逻辑变化

- 进货入库（+） / 采购退货（−） / 销售出库（−） / 客户退货（+） / 商品报损（−） / 商品报溢（+） / 盘点调整（±）这 7 类业务对库存的影响统一通过 `StockChangeService.apply(cmd)` 入口。
- 任何业务 Service 严禁直接 `UPDATE stock SET quantity = ...`。
- 单据保存（主表 + 明细 + N 次库存变更）在单事务内完成，任一步失败整单回滚。
- 单据作废（NORMAL → CANCELLED）写一笔反向 stock_log（delta 取反、reason 标"作废冲销"）。原 stock_log 不删、原明细不改。物理删除单据不允许。
- 库存数量并发使用 SQL 行锁（`SELECT ... FOR UPDATE`）。不引入分布式锁、不使用乐观锁。
- 库存报警仅为查询：`SELECT * FROM stock WHERE quantity < min_stock OR quantity > max_stock`，不维护独立 alarm 表，不做定时任务。
- 报表查询全部基于 `stock_log` + 单据主表聚合。`source_type` 是统计口径的唯一依据。

## 接口变化

第一阶段不新增接口。后续接口清单（按迭代）：

- 迭代 A：`/api/suppliers`、`/api/customers`。
- 迭代 B：`/api/purchases`、`/api/purchase-returns`、`/api/sales`、`/api/sales-returns`（每组各：列表 / 详情 / 创建 / 作废）。
- 迭代 C：`/api/stock-adjustments`（列表 / 详情 / 创建 / 作废）、`/api/stock-alarms`（查询）。
- 迭代 D：`/api/reports/purchase`、`/api/reports/sales`、`/api/reports/supplier`、`/api/reports/customer`；原 `/api/reports/*` 综合报表口径校准。
- 迭代 E：登录响应改为 `{ user, menus[], permissionCodes[] }`；`/api/roles`（含菜单权限绑定）、`/api/permissions`（列表）、`/api/system-logs`（查询）、`/api/auth/password`（修改）。

## 数据一致性

- `StockChangeService` 是库存一致性的承载点。所有数量变化与 stock_log 写入必须在同一事务、同一方法内完成。
- `stock_log` 扩展字段 `source_type` / `source_id` 是后续审计与报表的关键，不能省略。
- 旧 `stock_type` 字段保留兼容，但新写入由 `source_type` 主导。
- 字段对账：`stock.shelf_status` / `stock_log.stock_type` 在代码中已使用但正式 SQL 缺失；`restock_order` Mapper 存在但表不存在。迭代 A 必须先完成对账才能开始业务开发。
