# 后续迭代拆分

本任务（IT-004）只产出设计。业务升级实际开发由以下 5 个独立迭代承担。每个迭代以独立 task 形式登记到 `Document/3-项目开发/迭代优化/00-问题清单.md` 并拥有完整任务目录。

## 迭代依赖图

```
IT-004 业务模型与数据库设计（本任务）
   │
   ├── 迭代 A · 基础资料
   │     └── 迭代 B · 进货+销售 + StockChangeService
   │           ├── 迭代 C · 库存管理
   │           └── 迭代 D · 统计报表
   │     └── 迭代 E · 系统管理收尾
```

迭代 A 是所有后续迭代的前置；迭代 B 是 C / D 的前置；E 与 B/C/D 在 A 之后即可启动，与 B/C/D 并行无冲突。

## 迭代 A · 基础资料

**范围**

- 新建表：`supplier`、`customer`、`system_log`、`permission`、`role_permission`（仅建表，权限/日志业务逻辑留到 E）。
- 后端：SupplierController / CustomerController（CRUD）、相应 Service / Mapper。
- 前端：基础资料菜单组下三个页面（供应商管理、客户管理、商品管理）。
- 商品管理：菜单从原位置迁入"基础资料"，并把"分类管理"从独立菜单收纳到商品表单内部。
- 字段对账：`stock.shelf_status` / `stock_log.stock_type` / `restock_order` 的运行库 vs 正式 SQL 偏差全部修复。

**验收边界**

- supplier / customer 增删改查跑通；编码唯一约束生效。
- 商品页可在表单中选择分类，分类菜单已下线。
- 共 5 张新表（2 张主数据 supplier/customer + 3 张 system 类表 system_log/permission/role_permission）已 review 入正式 SQL，system 类表不接入任何业务逻辑。
- `Document/3-项目开发/market.sql` 与运行库字段完全一致。

**不算完成**

- 任何角色 / 权限 / 日志的业务逻辑落地（留给 E）。
- 旧 inbound/outbound 的退役（留给 E）。

## 迭代 B · 进货+销售 + StockChangeService

**这是本次升级最重的一块**。如果单次拉不下来，允许进一步拆 B1（进货）/ B2（销售），但不允许在 B1 跳过 StockChangeService。

**范围**

- 新建表：`purchase_order(+item)` / `purchase_return_order(+item)` / `sales_order(+item)` / `sales_return_order(+item)`，共 8 张。
- 扩展 `stock_log`：`source_type` / `source_id` / `reason` / `operator_id`。
- 后端：StockChangeService 单点入口；PurchaseService / PurchaseReturnService / SalesService / SalesReturnService；4 套 Controller / Mapper。
- 前端：8 个页面 — 进货入库、采购退货、进货单查询、采购退货查询、销售出库、客户退货、销售单查询、客户退货查询。
- 单据保存为单事务、明细多商品、保存即生效、可作废（反向冲销）。

**验收边界**

- 4 类单据可保存、可作废，作废写入反向 stock_log。
- 所有库存变更经过 StockChangeService 单点入口；行锁验证下并发不会负库存。
- 单据查询页支持时间段 / 对方主体 / 单据号过滤；分页正确。
- stock_log 所有新记录均带 source_type 与 source_id。
- 旧 InboundService / OutboundService 仍可工作但不再被新菜单调用（旧菜单要到 E 才下线）。

**不算完成**

- 报表口径校准（留给 D）。
- 报损报溢（留给 C）。
- 权限拦截（留给 E）。

## 迭代 C · 库存管理

**范围**

- 新建表：`stock_adjustment_order(+item)`。
- 后端：StockAdjustmentService（调用 StockChangeService，source_type=STOCK_LOSS / STOCK_OVERFLOW）。
- 前端：商品报损、商品报溢、报损报溢查询、库存报警；当前库存页保留。

**验收边界**

- 报损 / 报溢支持多商品明细。
- 库存报警页基于 `stock.quantity < min_stock OR stock.quantity > max_stock` 实时查询，无新增表。
- 报损报溢查询支持类型 / 时间 / 商品 / 操作人过滤。
- 报损 / 报溢的 stock_log 来源类型正确。

**不算完成**

- 独立 alarm 历史表或定时任务（不在本次升级范围）。

## 迭代 D · 统计报表

**范围**

- 后端：4 个新报表查询接口（采购统计 / 销售统计 / 供应商统计 / 客户统计），全部基于 stock_log + 单据主表聚合。
- 综合报表（原 ReportView）迁移到新模型，废弃对 inbound_order / outbound_order 的依赖。
- 首页 KPI 改为读新数据源。

**验收边界**

- 4 个新报表口径与 stock_log 一致；退货被正确扣减。
- 综合报表不再 SELECT inbound_order / outbound_order。
- 首页 KPI 数字与新业务一致。

**不算完成**

- 自定义报表、Excel 导出、复杂图表美化（按演示需求按需追加）。

## 迭代 E · 系统管理收尾

**范围**

- 后端：RoleController（角色 CRUD + 菜单权限绑定）、SystemLogController（日志查询）、PasswordController（修改）、登录接口扩展返回 `{ user, menus[], permissionCodes[] }`、Controller 注解级权限拦截、关键操作写入 system_log。
- 前端：角色管理页、系统日志查询页、密码修改弹窗（顶栏头像下拉）、菜单按 permissionCodes 动态渲染、路由守卫接入。
- 旧菜单下线：原"入库"/"出库"前端菜单移除（数据保留、表不删）。

**验收边界**

- 不同角色登录后看到不同菜单；越权访问被后端拒绝。
- system_log 记录登录、关键单据创建 / 作废、用户 / 角色 / 权限修改。
- 密码修改流程跑通；旧密码校验生效。
- 旧入库 / 出库菜单消失，相关路由不可达。

**不算完成**

- 按钮级权限、数据权限。

## 跨迭代风险登记

1. **字段对账（迭代 A 入口）**：`stock.shelf_status` / `stock_log.stock_type` / `restock_order` 在代码与正式 SQL 间存在偏差。**必须先对账，再开始 A 任务的业务开发**。
2. **迭代 B 体量风险**：8 表 + StockChangeService + 8 页面。允许在执行中拆为 B1/B2，但 StockChangeService 必须在第一阶段落地。
3. **菜单级权限的鸡生蛋**：登录接口必须一次性下发 `{ user, menus[], permissionCodes[] }`，前端持久化后再 mount router-view。E 的设计要点。
4. **演示数据自洽**：每次迭代结束补少量但完整的种子数据（供应商 5、客户 5、商品 30、采购 10、销售 15、退货各 3、报损报溢各 2），保证演示故事连贯。
