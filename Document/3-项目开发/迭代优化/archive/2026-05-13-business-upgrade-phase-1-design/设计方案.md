# 设计方案

## 问题定义

业务升级构想要把现有"商品、库存、入库、出库、盘点、报表、系统"结构，调整为更完整的进销存业务结构（首页 / 进货管理 / 销售管理 / 库存管理 / 统计报表 / 基础资料 / 系统管理）。分析文档判定整体属大改，应先做独立设计迭代，再拆分开发。

本任务是该业务升级的第一阶段，不实现功能，只产出业务模型、数据库草案、库存统一变更逻辑、菜单结构和后续迭代拆分。

## 现状分析

- 前端：旧菜单结构（商品/库存/入库/出库/盘点/报表/用户/系统）。
- 后端：product/category/stock/inbound/outbound/stockcheck/report/user/auth/system 模块齐备但围绕旧业务模型。
- 数据库：正式 SQL 已有 user/role/user_role/category/product/stock/inbound_order/outbound_order/stock_check/stock_log；缺供应商、客户、采购退货、客户退货、报损报溢、系统日志、权限表。
- 一致性偏差（来自 `Document/业务升级构想工作量与改造影响分析.md`）：`stock.shelf_status` / `stock_log.stock_type` 等字段在代码中已使用但正式 SQL 未定义；`restock_order` Mapper 存在但表不存在。**必须在后续任何开发迭代开始前完成字段对账**。

## 核心决策清单

| 决策 | 选择 |
| --- | --- |
| 单据数据模型 | 主表 + 明细表（采购、销售、采购退货、客户退货、报损报溢统一） |
| 退货结构 | 独立退货单表，可选关联原单 ID |
| 客户主数据 | 纳入本次升级 |
| 商品分类菜单 | 隐藏到商品页表单内部，不占独立菜单 |
| 角色权限 | 菜单级（permission + role_permission） |
| 系统日志 | 必做（system_log 表 + 查询页） |
| 单据状态机 | 单状态：保存即生效，仅 NORMAL / CANCELLED |
| 库存报警 | 查询页 + 阈值比较（`stock.quantity` vs `min_stock` / `max_stock`），不建表、不做定时任务 |
| 库存并发控制 | SQL 行锁（`SELECT ... FOR UPDATE`），不引入分布式锁 |
| 单据作废 | 反向冲销（写一笔取反 stock_log），不物理删除 |
| 报损报溢明细的金额字段 | unit_price / subtotal 可空 |

### 既有表改造

`stock_log` 必须扩展，承载库存统一变更：

```sql
ALTER TABLE stock_log
  ADD COLUMN source_type VARCHAR(32) COMMENT '来源单据类型 PURCHASE_IN/PURCHASE_RETURN_OUT/SALE_OUT/SALE_RETURN_IN/STOCK_LOSS/STOCK_OVERFLOW/STOCK_CHECK',
  ADD COLUMN source_id BIGINT COMMENT '来源单据ID',

  ADD COLUMN operator_id BIGINT;
```

旧 `stock_type` 字段保留兼容，新代码以 `source_type` 为准。

## §3 库存统一变更逻辑

### 单一入口 StockChangeService

所有库存数量变化只能通过一个服务方法发生：

```java
class StockChangeService {
  void apply(StockChangeCommand cmd);
}

class StockChangeCommand {
  Long productId;
  int delta;              // 正为增、负为减
  SourceType sourceType;  // 枚举
  Long sourceId;          // 来源单据 ID
  Long operatorId;
  String reason;
}
```

该方法承担：

1. 校验商品 / 库存存在性。
2. 更新 `stock.quantity`（行锁 `SELECT ... FOR UPDATE`，避免并发负库存）。
3. 写入 `stock_log`（source_type、source_id、operator_id、reason、变化前后数量）。

所有业务 Service（PurchaseService / SalesService / ReturnService / StockAdjustmentService / StockCheckService）必须调用此入口，禁止直接 `UPDATE stock SET quantity = ...`。

### 七种 source_type 与库存方向

| source_type | 触发 | 方向 | 来源表 |
| --- | --- | --- | --- |
| `PURCHASE_IN` | 进货入库 | + | purchase_order |
| `PURCHASE_RETURN_OUT` | 采购退货 | − | purchase_return_order |
| `SALE_OUT` | 销售出库 | − | sales_order |
| `SALE_RETURN_IN` | 客户退货 | + | sales_return_order |
| `STOCK_LOSS` | 商品报损 | − | stock_adjustment_order (type=LOSS) |
| `STOCK_OVERFLOW` | 商品报溢 | + | stock_adjustment_order (type=OVERFLOW) |
| `STOCK_CHECK` | 盘点调整 | ± | stock_check |

### 事务边界

一张单据（多商品）的保存是单事务：

```
BEGIN
  INSERT 主表
  INSERT 明细 × N
  FOR EACH 明细: StockChangeService.apply(...)
COMMIT
```

任一行失败 → 全单回滚。报表统计不会观察到"主表有单但 stock_log 没记"。

### 作废处理

单据 NORMAL → CANCELLED 时写一笔反向 stock_log（delta 取反、reason 标"作废冲销"）。不删除原 stock_log、不修改原明细。不允许物理删除单据。

### 报表口径副产物

有了统一 source_type，所有报表退化为聚合：

- 商品采购统计 = `stock_log WHERE source_type='PURCHASE_IN' GROUP BY product_id`
- 商品销售统计 = `stock_log WHERE source_type='SALE_OUT' GROUP BY product_id`
- 供应商统计 = `purchase_order GROUP BY supplier_id`（扣减 purchase_return_order）
- 客户统计 = `sales_order JOIN sales_return_order GROUP BY customer_id`

### 与现有 StockService 的关系

现有 `StockService.increaseStock / decreaseStock / adjustStock` 下沉为 `StockChangeService` 的内部细节，对外不再暴露。现有调用方（`InboundService` / `OutboundService`）随旧菜单下线一并退役。
