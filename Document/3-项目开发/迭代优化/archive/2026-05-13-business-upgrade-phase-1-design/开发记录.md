## 决策记录

通过 brainstorming 流程逐项确认（详见 `设计方案.md` 核心决策清单）：

- 单据采用主表 + 明细表（含报损报溢，统一模型）。
- 退货采用独立退货单表，可选关联原单 ID。
- 客户主数据纳入本次升级。
- 商品分类菜单收纳到商品表单内部。
- 角色权限做到菜单级（permission + role_permission）。
- 系统日志为必做项。
- 单据状态机仅 NORMAL / CANCELLED，保存即生效；作废采用反向冲销。
- 库存报警为查询型，不建表、不做定时任务。
- 库存并发用 SQL 行锁。
- 报损报溢明细的 unit_price / subtotal 允许为空。

## 问题与处理

- 业务升级整体大改，直接开发风险高 → 拆为 5 个后续开发迭代，各自有验收边界。
- 运行库 / 代码 / 正式 SQL 字段偏差（`stock.shelf_status` / `stock_log.stock_type` / `restock_order`）→ 列入迭代 A 入口动作，开发前必须对账。
- 库存口径混乱风险（采购入库、客户退货、报溢都被混记为普通入库）→ 通过 `StockChangeService` 单点入口 + `stock_log.source_type` 字段消解。
