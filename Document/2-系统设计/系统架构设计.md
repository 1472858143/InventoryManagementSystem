# 系统架构设计（前后端分离 + 业务领域模块化）

## 1 架构设计目标与原则

为满足超市库存管理系统在真实业务场景下的可维护性、可扩展性与云服务器部署需求，系统整体采用 **前后端分离架构**，后端基于 **Spring Boot** 构建 REST API，前端采用 **Vue** 实现用户界面，并部署于华为云服务器（ECS）环境。

在系统架构设计过程中，遵循以下原则：

- 前后端解耦原则：前端与后端独立开发、独立部署
- 单一职责原则（SRP）：模块、类与方法职责清晰
- 高内聚、低耦合原则：业务模块边界明确，减少模块间依赖
- 按业务领域拆分模块：以业务语义作为模块划分依据
- 核心业务集中控制原则：库存等关键业务能力采用单一入口设计

---

## 2 系统总体架构设计

### 2.1 前后端分离架构说明

系统采用典型的前后端分离架构模式：

- **前端（Vue）**
  - 负责页面展示与用户交互
  - 通过 HTTP(S) 调用后端 REST API
  - 构建产物为静态资源（dist），由 Nginx 统一托管

- **后端（Spring Boot）**
  - 对外提供 RESTful 风格接口
  - 负责业务逻辑处理、权限控制、数据持久化
  - 不承担页面渲染职责，仅返回 JSON 数据

在该架构下，传统 MVC 中的 View 层由前端 Vue 项目独立实现，后端 MVC 主要体现在 Controller 对请求的统一接入与响应。

---

### 2.2 云服务器部署架构（华为云 ECS）

系统部署于华为云 ECS，整体部署结构如下：

- **Nginx**
  - 托管前端 Vue 静态资源
  - 作为系统统一访问入口
  - 反向代理后端 Spring Boot API
  - 处理 HTTPS 证书与基础安全策略

- **Spring Boot 后端服务**
  - 提供 REST API 服务
  - 通过 Nginx 转发请求进行访问
  - 支持多环境配置（开发 / 生产）

- **数据库（MySQL）**
  - 存储系统业务数据
  - 可部署于 ECS 本地或使用云数据库服务

可选扩展组件：
- Redis：用于缓存、验证码、会话信息
- 对象存储：用于文件或图片上传
- 日志与监控：用于系统运行状态与操作行为记录

---

## 3 后端分层架构设计

后端系统内部采用分层架构，自上而下划分为以下层次：

### 3.1 接口层（Controller）

- 接收前端请求并完成参数绑定
- 调用业务层完成业务处理
- 返回统一格式的 JSON 响应结果
- 不直接编写业务逻辑

### 3.2 应用层（Service）

- 负责业务流程编排与事务控制
- 协调多个领域模块完成复杂业务用例
- 不直接依赖 HTTP 或数据库实现细节

### 3.3 领域层（Domain）

- 定义领域实体与业务规则
- 封装系统核心业务能力
- 对外提供领域服务（Domain Service）
- 作为业务一致性的核心保障层

### 3.4 持久化层（Repository / Mapper）

- 负责数据库访问操作（增删改查）
- 封装数据持久化细节
- 不包含业务逻辑判断

### 3.5 基础设施层（Common）

- 提供系统通用能力支持，包括：
  - 全局异常处理
  - 统一响应封装
  - 工具类
  - 权限认证与日志处理

---

## 4 业务领域模块化设计

系统采用 **按业务领域拆分模块** 的方式进行模块化设计，每个业务模块内部统一采用分层结构，实现“横向按业务拆分，纵向按功能分层”。

### 4.1 业务模块划分

系统主要业务模块包括：

- auth：认证与权限管理模块
- user：用户管理模块
- product：商品管理模块
- stock：库存管理模块（核心领域）
- inbound：入库管理模块
- outbound：出库管理模块
- stockcheck：库存盘点模块
- report：报表统计模块（只读）
- system：系统管理与配置模块

---

### 4.2 核心领域设计 —— 库存模块

库存模块作为系统的核心业务领域，负责库存数量与库存状态的维护。

为保证库存数据的一致性，系统采用 **库存变更单一入口设计原则**：

- 入库、出库、盘点等模块不得直接修改库存数据
- 所有库存数量变更操作必须统一调用库存领域服务
- 库存领域服务负责：
  - 校验库存变更的合法性
  - 执行库存数量的增减操作
  - 记录库存变更日志
  - 触发库存预警规则

该设计有效避免了库存数据被多处修改而导致的不一致问题。

---

## 5 模型对象设计与职责划分

为提升系统可维护性与接口稳定性，系统对模型对象进行明确拆分：

- **Entity**
  - 与数据库表一一对应
  - 仅用于数据持久化层

- **DTO（Data Transfer Object）**
  - 接收前端请求参数
  - 用于接口层与业务层之间的数据传递

- **VO（View Object）**
  - 封装返回前端的数据结构
  - 防止数据库实体对象直接暴露

- **Enum**
  - 定义业务状态与业务类型
  - 提升代码语义清晰度与可读性

## 7 架构约束规则

为保证系统架构的长期稳定性与可维护性，系统在设计阶段制定如下架构约束规则：

1. **Controller 层禁止直接访问 Repository 层**  
   所有数据访问操作必须通过 Service 层进行，防止接口层与持久化层直接耦合。

2. **Controller 层只接收 DTO 对象，只返回 VO 对象**  
   禁止将 Entity 直接暴露给前端，避免数据库结构与接口强耦合。

3. **Service 层负责业务流程控制与事务管理**  
   包括业务编排、事务边界控制以及跨模块业务协调。

4. **Domain 层负责核心业务规则与一致性控制**  
   所有关键业务规则应集中在领域层实现，避免业务逻辑分散。

5. **所有库存变更操作必须通过库存领域服务完成**  
   入库、出库、盘点等模块不得直接修改库存数据，以保证库存一致性。

6. **报表模块仅允许查询操作，不允许修改业务数据**  
   报表模块作为只读模块，避免对核心业务数据产生副作用。

7. **系统需区分开发环境与生产环境配置**  
   通过不同配置文件或环境变量实现多环境隔离，确保系统部署安全与稳定。
