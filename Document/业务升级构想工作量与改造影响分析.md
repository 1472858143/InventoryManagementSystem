# 业务升级构想工作量与改造影响分析

## 基本信息

- 分析日期：2026-05-13
- 项目背景：基于 Spring Boot + Vue 3 的超市库存管理系统
- 分析目标：仅评估工作量和改造影响，不进入代码实现
- 当前结论：本次业务升级整体属于大改，局部接近业务重构，不建议立即直接开发，应先作为独立迭代任务完成详细设计

## 总体判断

本次构想不是简单的菜单改名或页面增删。它把现有“商品、库存、入库、出库、盘点、报表、系统”结构，调整为更接近传统进销存系统的业务分层：

- 首页
- 进货管理
- 销售管理
- 库存管理
- 统计报表
- 基础资料
- 系统管理

其中“当前库存查询”“商品管理”“用户管理”“安全退出”“部分报表分析”可以复用现有能力；但供应商、客户、采购退货、客户退货、商品报损、商品报溢、系统日志、密码修改、角色管理完整化等内容需要新增数据结构、接口和页面。

因此工作量判断如下：

| 维度 | 判断 |
| --- | --- |
| 前端菜单与页面重组 | 中改 |
| 后端业务接口 | 大改 |
| 数据库结构 | 大改 |
| 库存统一变更逻辑 | 中到大改 |
| 统计报表 | 中到大改 |
| 文档同步 | 大改 |
| 是否接近重构 | 业务模型接近重构，技术框架不需要重构 |

## 现有模块可复用情况

| 目标功能 | 当前可复用内容 | 复用程度 | 说明 |
| --- | --- | --- | --- |
| 首页 | `frontend/src/views/home/HomeView.vue` | 高 | 可保留并按新菜单结构调整指标展示 |

| 进货入库 | `inbound` 模块、`inbound_order` | 中 | 可复用入库核心逻辑，但缺少供应商、单据号、金额、退货关联 |
| 销售出库 | `outbound` 模块、`outbound_order` | 中 | 可复用出库核心逻辑，但缺少客户、单据号、金额、销售退货关联 |
| 商品管理 | `product`、`category` 模块 | 高 | 可作为“基础资料-商品管理”继续使用 |
| 用户管理 | `user` 模块 | 中 | 已有用户管理基础，但系统管理还要求角色管理、密码修改等 |
| 安全退出 | `auth/logout` 和前端退出按钮 | 高 | 可菜单化或保留顶部操作 |
| 报表分析 | `report` 模块 | 中 | 已有库存概览、出入库趋势、销售热点等，统计口径需重做 |
| 库存报警 | 现有 `min_stock`、`max_stock`、首页 warning 统计 | 中 | 可复用阈值字段，但需要正式报警页面和查询接口 |
| 盘点差异 | `stock_check` 模块 | 中 | 可参考，但“报损/报溢”不应完全等同于盘点 |

## 需要新增或明显调整的数据库表

当前正式 SQL 主要包括：`user`、`role`、`user_role`、`category`、`product`、`stock`、`inbound_order`、`outbound_order`、`stock_check`、`stock_log`。

建议新增或调整如下：

| 业务对象 | 建议表 | 必要性 | 说明 |
| --- | --- | --- | --- |
| 供应商管理 | `supplier` | 必需 | 进货管理和供应商统计需要供应商主数据 |
| 客户管理 | `customer` | 必需 | 销售管理和客户统计需要客户主数据 |
| 进货单据增强 | `purchase_order` 或扩展 `inbound_order` | 建议新增 | 现有入库单过轻，缺少供应商、单据编号、金额、状态 |
| 进货明细 | `purchase_order_item` | 视需求 | 如果一张单据包含多个商品，必须新增明细表 |
| 退货出库 | `purchase_return_order` / `purchase_order` 增加类型 | 必需 | 采购退货与普通出库业务语义不同 |
| 销售单据增强 | `sales_order` 或扩展 `outbound_order` | 建议新增 | 现有出库单缺少客户、金额、单据编号、状态 |
| 销售明细 | `sales_order_item` | 视需求 | 如果销售单支持多商品，必须新增明细表 |
| 客户退货 | `sales_return_order` / `sales_order` 增加类型 | 必需 | 客户退货会增加库存，不能简单等同于入库 |
| 报损报溢 | `stock_adjustment_order` | 必需 | 记录报损、报溢类型、原因、操作人、审核状态 |
| 系统日志 | `system_log` | 必需 | 系统管理要求系统日志时需要持久化操作记录 |
| 密码修改记录 | 可并入 `system_log` | 可选 | 不一定单独建表，但应记录关键安全操作 |
| 角色权限 | `permission`、`role_permission` | 可选 | 如果只是角色基础管理可不建；如果要菜单/按钮权限则需要 |

重要提醒：当前代码中 `StockMapper.xml`、`ReportMapper.xml`、`StockLogMapper.xml` 已使用 `stock.shelf_status`、`stock_log.stock_type` 等字段，但 `sql/market.sql` 中未看到这些字段定义；代码中还存在 `restock_order` Mapper，但正式 SQL 未见对应表。正式开发前应先核对运行库、正式 SQL 和文档 SQL 的一致性。

## 只需要前端菜单和页面调整的内容

这些内容可以先作为低风险改造处理：

| 功能 | 改造内容 | 工作量 |
| --- | --- | --- |
| 报表分析更名为统计报表 | 菜单标题、路由标题、页面标题调整 | 小 |
| 系统更名为系统管理 | 菜单分组名称调整 | 小 |
| 商品管理移动到基础资料 | 菜单重组，复用现有商品页 | 小到中 |
| 当前库存查询多处出现 | 可复用同一个库存查询页面，多个菜单指向同一路由或封装公共组件 | 小到中 |
| 安全退出 | 现有退出登录按钮可调整到系统管理菜单或保留顶部按钮 | 小 |

但如果要求每个菜单都拥有独立页面、独立查询条件、独立权限，则会从小改上升为中改。

## 需要新增后端接口的内容

| 模块 | 新增接口方向 | 说明 |
| --- | --- | --- |
| 进货管理 | 进货入库、退货出库、进货单查询、退货单查询 | 现有入库只能覆盖一部分，需要供应商、单据号、金额、退货类型 |
| 销售管理 | 销售出库、客户退货、销售单查询、客户退货查询 | 现有出库只能覆盖一部分，需要客户、销售金额、退货类型 |
| 库存管理 | 商品报损、商品报溢、库存报警、报损报溢查询 | 需要新增调整单和库存变更类型 |
| 统计报表 | 供应商统计、客户统计、商品采购统计、商品销售统计 | 现有报表口径不足，需要基于新采购/销售/退货数据统计 |
| 基础资料 | 供应商管理 | 当前没有供应商模块 |
| 系统管理 | 角色管理、系统日志、密码修改 | 用户管理已有基础，但角色管理和日志不完整 |

## 对库存统一变更逻辑的影响

这是本次升级中最关键的影响点。

现有库存核心能力集中在 `StockService`：

- `increaseStock`：入库增加库存
- `decreaseStock`：出库减少库存
- `adjustStock`：盘点调整库存
- `stock_log`：记录库存变化

新业务下，所有库存变化都应统一进入库存变更逻辑：

| 业务动作 | 库存方向 | 建议变更类型 |
| --- | --- | --- |
| 进货入库 | 增加 | `PURCHASE_IN` |
| 退货出库 | 减少 | `PURCHASE_RETURN_OUT` |
| 销售出库 | 减少 | `SALE_OUT` |
| 客户退货 | 增加 | `SALE_RETURN_IN` |
| 商品报损 | 减少 | `STOCK_LOSS` |
| 商品报溢 | 增加 | `STOCK_OVERFLOW` |
| 盘点调整 | 增加或减少 | `STOCK_CHECK` |

建议把现有 `INBOUND / OUTBOUND / CHECK / SHELF_STATUS` 扩展成更清晰的业务类型，并让 `stock_log` 记录来源单据类型、来源单据 ID、操作人、原因。否则后续统计、审计、回溯会困难。

如果只是在各业务 Service 中直接调用 `increaseStock` / `decreaseStock`，短期可以跑通，但长期会出现统计口径混乱：采购入库、客户退货、报溢都会被当成普通入库；销售出库、采购退货、报损都会被当成普通出库。

## 分模块工作量评估

| 模块 | 工作量等级 | 原因 |
| --- | --- | --- |
| 首页 | 小到中 | 可以复用现有首页，但指标口径需跟新业务同步 |
| 进货管理 | 中到大 | 进货入库可复用，但退货、供应商、单据查询和统计字段需要补齐 |
| 销售管理 | 中到大 | 销售出库可复用出库逻辑，但客户退货和客户维度需要新增 |
| 库存管理 | 中到大 | 报损报溢、报警、查询需要新增，且影响库存统一变更逻辑 |
| 统计报表 | 大 | 报表口径依赖采购、销售、客户、供应商新模型 |
| 基础资料 | 中 | 商品可复用，供应商需新增；是否保留分类管理需要确认 |
| 系统管理 | 中到大 | 用户可复用，角色管理、系统日志、密码修改需要补齐 |
| 数据库设计 | 大 | 需要新增多张业务表，并统一单据和库存日志关系 |
| 文档同步 | 大 | 项目书、需求分析、系统设计、数据库设计和正式 SQL 都会变化 |

整体估算：如果按“页面可用 + 后端接口可用 + 数据库一致 + 基本测试”标准，建议拆成 4 到 6 个迭代处理。若一次性完成全部内容，风险接近重构级别。

## 建议的实施拆分

不建议立即按完整构想直接开发。建议先建立独立迭代任务，完成业务设计和数据库草案，再进入代码实现。

推荐拆分如下：

1. 业务模型设计迭代
   - 明确采购、销售、退货、报损报溢是否采用“单据主表 + 明细表”
   - 明确供应商、客户、角色权限的最小字段
   - 明确哪些旧功能保留、隐藏或删除

2. 基础资料迭代
   - 供应商管理
   - 商品管理菜单迁移
   - 可选：客户管理若不放在基础资料，也要先建客户主数据

3. 进货与销售迭代
   - 进货入库、退货出库
   - 销售出库、客户退货
   - 单据查询
   - 接入统一库存变更

4. 库存管理迭代
   - 报损、报溢
   - 库存报警
   - 报损报溢查询
   - 当前库存查询复用

5. 统计报表迭代
   - 供应商统计
   - 客户统计
   - 商品采购统计
   - 商品销售统计
   - 原报表分析重命名和口径调整

6. 系统管理迭代
   - 角色管理
   - 系统日志
   - 密码修改
   - 安全退出菜单化

## 结论

本次升级不是小改，也不适合在当前登录系统升级 task 中直接合并开发。它应作为一个独立业务升级迭代进行设计，优先完成数据模型和统一库存变更方案。

最稳妥的策略是：

- 第一阶段只做设计：模块边界、数据库草案、库存变更类型、菜单结构。
- 第二阶段先做基础资料和核心单据：供应商、客户、采购、销售。
- 第三阶段再做库存报损报溢和统计报表。
- 最后再同步正式项目书、需求分析、系统设计和正式 SQL。

如果只是为了演示效果，可以先做菜单重组和页面命名调整；但如果要让业务闭环真实成立，必须补数据库和后端接口，整体应按大改处理。
