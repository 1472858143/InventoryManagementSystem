# 前端开发规划

## 1. 文档目的

本文档用于作为库存管理系统前端项目的开发参考手册，明确前端开发范围、目录职责、开发顺序、页面规范、接口联调流程、异常处理要求和打包验证标准。

本文档只面向开发执行，不作为答辩说明文档使用。后续前端页面开发、接口联调和阶段性收尾应优先参考本文档推进。

当前前端开发必须遵守以下原则：

- 以项目书为唯一基准
- 不扩展项目书之外的业务模块
- 不改变既有技术方案
- 不引入复杂架构
- 不把后端业务规则搬到前端实现

## 2. 项目约束

### 2.1 技术方案约束

前端技术方案固定如下：

- 前端框架：`Vue 3`
- 构建工具：`Vite`
- 前端语言：`JavaScript`
- 前后端架构：前后端分离
- 前端运行环境：`Node.js 24.14.0`
- 后续部署方式：前端打包生成静态资源，可通过 `Nginx` 部署并反向代理后端接口

当前阶段不引入以下复杂方案：

- 微前端
- SSR
- 复杂权限系统
- 项目书未要求的低代码配置
- 项目书未要求的供应商、采购等模块

### 2.2 模块范围约束

前端只实现项目书中明确的模块：

- `auth`
- `user`
- `product`
- `stock`
- `inbound`
- `outbound`
- `stockcheck`
- `report`
- `system`

禁止新增项目书之外的模块，例如：

- `purchase`
- `supplier`
- `customer`
- `finance`
- 其他未在项目书中定义的业务模块

### 2.3 库存业务约束

前端不参与库存计算或库存修改。

要求：

- 所有库存变化必须由后端 `stock` 模块统一处理
- 前端只负责调用后端接口并展示结果
- 前端不得自行计算库存增减后的最终值并写回
- 前端不得提供绕过后端 `stock` 模块的库存修改入口
- 入库、出库、盘点页面只能调用对应业务接口
- 库存真实变化由后端 `stock` 模块完成

### 2.4 report 模块约束

`report` 模块只读。

要求：

- 页面只展示统计或查询结果
- 不允许新增操作
- 不允许修改操作
- 不允许删除操作
- 不允许提供库存调整入口

### 2.5 system 模块约束

`system` 模块保持轻量化。

当前阶段只做系统说明、运行说明、账号入口说明等轻量展示，不实现复杂系统配置、日志管理、参数配置、权限配置等功能。

### 2.6 安全约束

前端必须遵守以下安全边界：

- 不保存明文密码
- 不展示密码或密码哈希字段
- token 统一通过请求头传递
- 生产环境前端提交原始密码时，认证链路应通过 `HTTPS`
- 未登录或 token 失效时应跳转登录页

## 3. 前端项目结构说明

当前仓库根目录已经存在前端工程：

- `frontend/`

前端工程采用项目书要求的 Vue 3 结构，当前基础目录如下：

```text
frontend
├─ public/
├─ src/
│  ├─ api/
│  ├─ assets/
│  ├─ components/
│  ├─ layout/
│  ├─ router/
│  ├─ stores/
│  ├─ utils/
│  ├─ views/
│  │  ├─ login/
│  │  ├─ user/
│  │  ├─ product/
│  │  ├─ stock/
│  │  ├─ inbound/
│  │  ├─ outbound/
│  │  ├─ stockcheck/
│  │  ├─ report/
│  │  └─ system/
│  ├─ App.vue
│  └─ main.js
├─ package.json

```

当前阶段不需要重新创建项目骨架，应在现有结构基础上按开发流程逐步补齐页面、请求、路由和联调记录。

### 3.1 前端结构与后端模块映射

前端 `views` 目录必须与后端模块一一对应，避免前端结构与后端接口脱节。

映射关系如下：

| 前端页面目录 | 对应后端模块 | 说明 |
| --- | --- | --- |
| `views/login` | `auth` | 登录与认证入口 |
| `views/user` | `user` | 用户管理 |
| `views/product` | `product` | 商品管理 |
| `views/stock` | `stock` | 当前库存与库存上下限管理 |
| `views/inbound` | `inbound` | 入库记录与新增入库 |
| `views/outbound` | `outbound` | 出库记录与新增出库 |
| `views/stockcheck` | `stockcheck` | 盘点记录与新增盘点 |
| `views/report` | `report` | 报表展示，只读 |
| `views/system` | `system` | 系统信息展示，轻量化 |

## 4. 目录职责说明

### 4.1 api

`api` 目录只负责封装后端接口请求。

要求：

- 按后端模块拆分 API 文件
- 所有请求必须通过 `api` 层封装
- 必须统一使用请求工具
- 不允许在页面中直接写接口地址
- 不在 API 文件中写页面展示逻辑
- 不在 API 文件中处理库存规则

建议 API 文件按模块组织：

- `auth`
- `user`
- `product`
- `stock`
- `inbound`
- `outbound`
- `stockcheck`
- `report`
- `system`

### 4.2 router

`router` 目录负责路由配置和访问控制。

要求：

- 配置登录页路由
- 配置后台主布局下的业务页面路由
- 处理未登录访问跳转
- 不在路由中写接口业务逻辑
- 不在路由中处理库存规则

建议路由路径：

- `/login`
- `/users`
- `/products`
- `/stocks`
- `/inbounds`
- `/outbounds`
- `/stockchecks`
- `/reports`
- `/system`

### 4.3 stores

`stores` 目录负责前端状态管理。

当前阶段重点维护：

- token
- 当前用户信息
- 登录状态
- 菜单展开状态

要求：

- 不保存明文密码
- token 持久化策略保持简单
- 页面刷新后可恢复基础登录状态
- 不在 store 中实现后端业务规则

### 4.4 layout

`layout` 目录负责后台管理系统整体框架。

建议包含：

- 顶部栏
- 侧边栏
- 主内容区域
- 退出登录入口

要求：

- 不承载具体业务模块逻辑
- 不直接调用业务接口
- 只负责后台通用布局

### 4.5 components

`components` 目录存放可复用组件。

当前阶段可按需要提取：

- 通用表格区域
- 通用弹窗
- 通用表单按钮区
- 空状态提示
- 加载状态提示

要求：

- 不为了抽象而提前抽象
- 优先保证页面功能清晰可用
- 组件只处理展示和交互，不处理复杂业务规则

### 4.6 views

`views` 目录负责具体业务页面。

要求：

- 页面目录与后端模块对应
- 页面负责展示、表单输入、调用 API 和显示结果
- 不直接拼接后端 URL
- 不绕过统一请求工具
- 不在页面中实现库存计算规则
- 不在页面中实现后端业务校验规则

### 4.7 utils

`utils` 目录负责通用工具方法。

当前阶段可包含：

- token 读写工具
- 日期格式化工具
- 请求错误消息处理工具

要求：

- 工具函数保持简单
- 不引入复杂通用框架
- 不把业务模块逻辑塞入工具层

## 5. 开发流程

前端开发必须按以下阶段推进，顺序不可打乱。

### 5.1 阶段一：基础能力搭建

目标：完成前端项目运行所需的基础能力，为后续页面开发提供统一框架。

开发内容：

1. 确认项目结构
2. 清理默认示例页面
3. 搭建后台主布局 `layout`
4. 配置基础路由 `router`
5. 封装统一请求工具
6. 建立 token 读写工具
7. 建立基础错误提示能力

完成标准：

- 前端项目可正常启动
- 浏览器可访问登录页
- 后台主布局可正常显示
- 侧边栏菜单可跳转
- 请求工具具备统一 baseURL、token 注入、响应解析能力
- token 可写入、读取、清理

### 5.2 阶段二：认证体系

目标：完成登录、token 存储、登录态恢复和路由守卫。

开发内容：

1. 实现登录页
2. 封装登录接口
3. 登录成功后保存 token
4. 登录成功后保存必要用户信息
5. 配置路由守卫
6. 未登录访问业务页面时跳转登录页
7. 退出登录时清理 token 与用户状态

完成标准：

- 登录接口可正常调用
- 登录成功后可进入后台主界面
- 登录失败时展示后端 `message`
- 页面刷新后可恢复基础登录状态
- 未登录访问业务页会跳转到登录页
- 退出登录后不能继续访问业务页
- 前端不保存明文密码

### 5.3 阶段三：核心业务页面

目标：完成系统基础数据和库存基础页面，为入库、出库、盘点页面提供前置数据支撑。

开发顺序固定为：

1. `user`
2. `product`
3. `stock`

#### 5.3.1 user 页面

开发内容：

- 用户列表
- 新增用户表单
- 启用/禁用用户
- 角色编码展示

完成标准：

- 可查询用户列表
- 可新增用户
- 可修改用户状态
- 不展示 password 或密码哈希
- 参数错误时展示后端 `message`

#### 5.3.2 product 页面

开发内容：

- 商品列表
- 新增商品表单
- 上架/下架商品

完成标准：

- 可查询商品列表
- 可新增商品
- 可修改商品状态
- 能展示商品编码、名称、分类、价格、状态
- 商品编码重复、价格非法等错误可正常提示

#### 5.3.3 stock 页面

开发内容：

- 库存列表
- 单个商品库存查看
- 库存上下限维护

完成标准：

- 可查询库存列表
- 可查看单个商品库存
- 可维护库存上下限
- 页面不提供直接手工修改库存数量的入口
- 上下限非法时展示后端 `message`

### 5.4 阶段四：库存相关业务

目标：完成入库、出库、盘点页面。库存数量变化必须通过后端对应业务接口触发，并由后端 `stock` 模块统一处理。

开发顺序固定为：

1. `inbound`
2. `outbound`
3. `stockcheck`

#### 5.4.1 inbound 页面

开发内容：

- 入库记录列表
- 新增入库表单

完成标准：

- 可查询入库记录
- 可新增入库记录
- 新增后库存数量由后端正确增加
- 前端不直接修改库存数量
- 商品不存在、库存记录不存在、数量非法时展示后端 `message`

#### 5.4.2 outbound 页面

开发内容：

- 出库记录列表
- 新增出库表单

完成标准：

- 可查询出库记录
- 可新增出库记录
- 新增后库存数量由后端正确扣减
- 库存不足时展示后端 `message`
- 前端不自行判断最终库存是否合法

#### 5.4.3 stockcheck 页面

开发内容：

- 盘点记录列表
- 新增盘点表单
- 差异结果展示

完成标准：

- 可查询盘点记录
- 可新增盘点记录
- 可展示系统库存、实际库存、差异数量
- 新增后库存数量由后端调整为实际库存
- 前端不自行执行库存调整

### 5.5 阶段五：展示类页面

目标：完成只读或轻量展示页面。

开发顺序固定为：

1. `report`
2. `system`

#### 5.5.1 report 页面

开发内容：

- 库存相关统计展示
- 入库、出库、盘点数据汇总展示

完成标准：

- 报表只查询数据
- 不写业务数据
- 不提供新增、修改、删除操作
- 不提供库存调整入口

#### 5.5.2 system 页面

开发内容：

- 系统说明
- 当前账号信息入口
- 运行说明或基础系统信息

完成标准：

- 页面可访问
- 内容与项目书中 `system` 模块轻量化定位一致
- 不实现复杂系统配置
- 不实现复杂日志管理

## 6. 页面开发规范

### 6.1 页面基本结构

每个业务页面应按以下结构组织：

1. 列表区域
2. 操作按钮区域
3. 新增或编辑表单
4. 异常提示
5. 空状态展示
6. loading 状态

### 6.2 查询能力

列表页面必须优先实现查询能力。

要求：

- 页面进入时自动查询
- 查询失败时展示后端 `message`
- 查询结果为空时展示空状态
- 不在前端补造假数据

### 6.3 新增能力

涉及新增的页面应提供新增表单。

要求：

- 表单字段与后端 DTO 对齐
- 提交前做基础必填提示
- 业务校验结果以后端返回为准
- 提交成功后刷新列表
- 提交中禁止重复提交

### 6.4 修改或状态操作

涉及修改或状态操作的页面应保持操作范围清晰。

要求：

- 只实现后端已提供的修改或状态接口
- 不扩展后端未提供的编辑能力
- 操作成功后刷新列表或局部更新
- 操作失败时展示后端 `message`

### 6.5 异常处理

页面必须处理以下异常场景：

- 参数错误
- 业务错误
- 未登录或 token 失效
- 网络异常
- 返回数据为空

### 6.6 禁止事项

页面中禁止：

- 直接写后端接口地址
- 绕过 `api` 层调用接口
- 写复杂业务逻辑
- 处理库存计算规则
- 手工修改库存数量
- 实现项目书之外的业务功能

## 7. 接口与联调规范

### 7.1 接口调用规范

所有接口调用必须遵守以下规则：

- 所有请求必须通过 `api` 层封装
- 所有请求必须使用统一请求工具
- 页面中不允许直接写接口地址
- token 统一通过请求头传递
- 请求头格式为 `Authorization: Bearer <token>`
- 不在页面中重复写 token 拼接逻辑

### 7.2 响应结构处理

后端统一返回结构为：

```json
{
  "code": 0,
  "message": "success",
  "data": {}
}
```

前端处理规则：

- `code = 0` 表示成功，使用 `data`
- `code != 0` 表示失败，优先展示 `message`
- 不直接依赖 HTTP 状态码判断业务成功失败
- JSON 格式错误、参数错误、业务异常都应展示后端返回的 `message`

### 7.3 联调总顺序

接口联调按以下顺序推进：

1. 登录获取 token
2. 验证统一请求封装
3. 联调 `user`
4. 联调 `product`
5. 联调 `stock`
6. 联调 `inbound`
7. 联调 `outbound`
8. 联调 `stockcheck`
9. 联调 `report`
10. 联调 `system`

### 7.4 单个模块联调顺序

每个模块必须按以下顺序联调：

1. 查询接口
2. 新增接口
3. 修改或状态接口
4. 异常场景
5. 未登录场景

如果某模块没有新增、修改或状态接口，则跳过不存在的接口，不额外扩展。

### 7.5 联调执行要求

要求：

- 每个页面开发完成后必须立即联调
- 不允许所有页面写完后再统一联调
- 每个模块先测查询，再测写操作
- 每完成一个页面必须记录联调结论
- 联调发现后端问题时记录接口、请求参数、响应结果

## 8. 异常处理规范

### 8.1 错误提示

前端应优先展示后端返回的 `message` 字段。

示例：

- 商品不存在
- 库存记录不存在
- 库存数量非法
- 商品ID不能为空
- 实际库存不能小于0
- 未登录或认证失败

要求：

- 不吞掉后端错误信息
- 不统一显示含糊的“操作失败”
- 无后端 message 时再显示通用错误提示

### 8.2 空状态提示

列表为空时必须显示明确提示。

建议提示：

- 暂无数据
- 暂无用户记录
- 暂无商品记录
- 暂无库存记录
- 暂无入库记录
- 暂无出库记录
- 暂无盘点记录

### 8.3 loading 状态

请求进行中应显示 loading 状态。

要求：

- 列表查询时显示加载状态
- 表单提交时按钮进入 loading 状态
- 请求结束后恢复

### 8.4 防重复提交

写操作必须防止重复提交。

要求：

- 提交中禁用提交按钮
- 请求完成后恢复按钮
- 提交成功后关闭表单或清空表单
- 提交失败时保留用户输入，便于修正

### 8.5 未登录处理

当后端返回 `401` 时：

- 清理本地 token
- 清理当前用户状态
- 跳转登录页
- 提示用户重新登录

## 9. 打包验证

### 9.1 打包目标

确认前端项目能够生成可部署静态资源，为后续部署和验收做准备。

### 9.2 打包前检查

打包前应确认：

- 所有页面可正常访问
- 登录和退出正常
- 主要接口联调完成
- 异常提示可用
- 空状态展示可用
- 没有明显控制台错误

### 9.3 打包任务

执行前端构建命令后，需要检查：

- 是否生成 `dist`
- 构建过程是否报错
- 静态资源路径是否正常
- 页面刷新是否存在路由问题

### 9.4 完成标准

- 前端项目可正常打包
